# Skill Discovery — 能力沉淀器

## 触发约束

此 skill **仅通过显式调用触发**。

### ⛔ 不触发的场景
- 用户提到"发现模式"、"提炼能力"等但未提及 skill-discovery
- 完成任务后的"顺便"扫描
- 用户未显式引用 @skill-discovery

### ✅ 触发条件
必须同时满足：
1. 用户明确说出"使用 skill-discovery"或"启动 skill-discovery"，或显式引用 @skill-discovery
2. 用户提供了明确需求或上下文

---

## 问题

在使用 agent 的过程中，我们积累了大量隐性知识和工作模式：

1. **反复纠正同一个行为**——每次都教，但从不沉淀
2. **同类修复反复出现**——同样的 bug 修了又修
3. **工作模式被遗忘**——某个流程跑通了，下次又从头摸索
4. **Anti-pattern 反复踩**——文档里写了但 agent 不看

**如果不主动提炼，这些能力就会流失。** 这个 skill 就是让"从经验中提炼能力"这件事本身变得可重复、系统化。

---

## 核心原则

```
扫描 → 聚类 → 评估 → 检查覆盖度 → 草案 → 用户确认
```

每个阶段都是必需的：
- **扫描**不是罗列，而是识别模式
- **聚类**不是去重，而是发现共性
- **评估**不是打分，而是判断杠杆
- **检查覆盖度**在确定候选后，判断新建还是修改现有 skill
- **草案**不是笔记，而是可用的 SKILL.md
- **确认**由用户审查决定，skill 不自动写入任何文件

---

## 执行流程

### Phase 1: 信号收集

按以下顺序扫描项目，收集原始信号：

#### 1a. Git 提交模式

```bash
# 提交类型分布
git log --oneline -100

# 修复模式聚类
git log --all --oneline --grep="fix" --grep="修复" --grep="bug" -i -50

# 重构模式
git log --all --oneline --grep="refactor" --grep="重构" -i -50

# 同类问题反复修？
git log --all --oneline --grep="yaml" --grep="schema" --grep="格式" -i -30
```

提取信号：
- **同类修复反复出现** → 预防性 skill 候选
- **同一模块频繁重构** → 架构约束 skill 候选
- **跨文件同步修改** → 一致性 skill 候选

#### 1b. 反馈归档目录

扫描项目反馈归档目录（常见路径：`docs/feedback/archive/`、用户指定路径）：

提取信号：
- **反复出现的问题类型** → 预防性 skill 候选（同类问题归档 ≥3 条）
- **高频修复的模块/文件** → 针对性 skill 候选（同一文件多次出现在归档中）
- **用户习惯性踩的坑** → 护栏 skill 候选

#### 1c. Agent 对话模式

扫描项目的 agent transcripts 目录（路径因编辑器不同）：
- Claude Code: 查阅对应文档
- Cursor: `~/.cursor/projects/*/agent-transcripts/*.jsonl`
- 用户指定路径

提取信号：
- **用户反复下达的同类指令** → 自动化 skill 候选
- **用户反复纠正的 agent 行为** → 护栏 skill 候选
- **agent 反复犯的同类错误** → 约束 skill 候选
- **"你应该先读..."类提醒反复出现** → 流程 skill 候选

---

### Phase 2: 模式聚类

将 Phase 1 收集的信号按语义分组：

| 聚类方法 | 说明 |
|---------|------|
| **按触发条件** | "当 X 发生时，总是做 Y" |
| **按影响范围** | 全局 / 项目级 / 模块级 |
| **按出现频率** | ≥2 次才考虑，≥5 次优先 |
| **按修复代价** | 每次修复耗时 × 出现次数 = 累计代价 |

聚类后过滤掉：
- 单次事件（不具复用性）
- 过于特定（只在某一次 commit 中出现）
- 成本过高（需要人工判断且无法标准化）

---

### Phase 3: 价值评估

按直觉判断优先级：

| 优先级 | 判断标准 |
|--------|----------|
| **高优先** | 反复出现（≥3次）且每次处理耗时 > 15分钟 |
| **中优先** | 出现过 2-3 次，有明显复用价值 |
| **低优先** | 单次事件或过于特定，不值得提炼 |

只处理高优先和中优先候选。低优先候选记录在案但不生成草案。

---

### Phase 4: 覆盖度检查

在确定候选模式后，检查现有 skill 是否已覆盖，决定行动方向。

#### 4a. 扫描现有 skill

扫描目标：
- **项目级**：`.claude/skills/`、`.cursor/rules/`、`AGENTS.md`
- **全局级**：`~/.claude/skills/`、`~/.cursor/skills/`

#### 4b. 判断覆盖情况

对每个候选模式，判断与现有 skill 的关系：

| 覆盖情况 | 判断标准 | 行动 |
|---------|---------|------|
| **完全覆盖** | 候选模式与现有 skill 功能完全一致 | 记录"已存在 skill: X"，不生成草案 |
| **部分覆盖** | 现有 skill 解决了部分问题，但仍有未覆盖的功能 | 建议修改现有 skill，标注需扩展的功能 |
| **无覆盖** | 无相关 skill 或完全不匹配 | 生成新 skill 草案 |

#### 4c. 输出覆盖度报告

```
--- 覆盖度检查 ---
候选: {skill-name}
覆盖情况: {完全覆盖/部分覆盖/无覆盖}
相关 skill: {skill 名称 或 "无"}
建议行动: {不生成草案/修改现有 skill/新建 skill}
```

---

### Phase 5: 草案生成策略

首先检查 my-create-skill 是否可用：

扫描 skill 目录：
- `.claude/skills/my-create-skill/SKILL.md`
- `.cursor/skills/my-create-skill/SKILL.md`

**如果可用**：
- 输出候选清单（名称、触发条件、优先级）
- 建议用户：`使用 my-create-skill 创建 {skill-name}`
- 本 skill 不生成完整草案，my-create-skill 会处理格式规范

**如果不可用**：
- 本 skill 生成简化草案框架（不含完整触发约束段落）
- 提醒用户：草案格式可能不完整，建议后续校验

简化草案格式：

```yaml
---
name: {skill-name}
description: {1句话描述，说明用途}
---

# {Skill 标题}

## 问题
{这个 skill 解决什么问题，用真实案例说明}

## 核心原则
{1-3 条核心原则}

## 执行流程
{步骤化的操作指南}
```

**草案质量要求**：
- MUST 包含真实案例（从扫描数据中提取）
- MUST 包含可执行的步骤
- NEVER 编造未扫描到的模式
- NEVER 生成空模板或占位符

---

### Phase 6: 输出与交付

生成 **能力发现报告**（文本形式）：

```
=== 能力发现报告 ===

扫描范围：git（30天）、transcripts（用户指定）
发现候选：X 个

--- 高优先候选 ---
1. {skill-name}
   触发条件：用户说"..."
   模式来源：git fix 反复出现 5 次

--- 中优先候选 ---
...

--- 建议行动 ---
my-create-skill 可用，建议执行：
  使用 my-create-skill 创建 {skill-name}
```

**输出选项**：
- 默认输出文本报告（对话中展示）
- 用户明确要求时可写入文件：
  - 报告：用户指定路径
  - 草案：`.claude/skills/{skill-name}/SKILL.md`（仅当 my-create-skill 不可用时）

---

### Phase 7: 用户审核确认

展示完整报告后，进入审核阶段：

**审核提示格式**：
```
共发现 X 个 skill 候选

--- 候选 1 ---
名称: {skill-name}
优先级: 高
触发条件: {触发条件}
模式来源: {扫描来源}
草案预览: {简要内容}

--- 候选 2 ---
...

请确认以上候选：
- 输入序号可查看/修改单个候选详情（如输入"1"查看候选1）
- 输入"接受"确认全部候选，后续可用 my-create-skill 创建
- 输入"取消"中止，不输出任何草案
```

**用户操作**：
- **序号（如 1、2）**：查看指定候选详情，可调整名称/触发条件/优先级
- **接受**：确认候选清单，输出完整草案框架
- **取消**：中止，不输出草案

**注意**：
- 此阶段只输出文本草案，不写入文件
- 用户需显式说"写到 XXX"才执行写入

---

## 硬约束

- **MUST 只读扫描** — 扫描阶段不修改任何项目数据
- **MUST 输出文本** — 报告和草案以对话文本形式输出，不默认写文件
- **MUST 用户指定写入路径** — 只有用户明确说"写到 XXX"时才写入
- **MUST 遵循格式** — 草案与用户已有全局 skill 格式一致
- **NEVER 自动写入文件** — 不写入 skill 目录、不写入临时文件、不写入任何位置
- **NEVER 编造模式** — 不凭空想象可能的 skill
- **NEVER 自动注册** — skill 的创建、注册、提交全部由用户手动完成
- **NEVER 扫描敏感数据** — 跳过 credentials、tokens、.env 等文件
- **NEVER 在扫描失败时静默跳过** — 明确报告哪些数据源不可用
