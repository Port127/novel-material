# Skill Discovery — 能力沉淀器

## 触发约束

此 skill **仅通过显式调用触发**。

### ⛔ 不触发的场景
- 用户提到"发现模式"、"提炼能力"等但未提及 skill-discovery
- 完成任务后的"顺便"扫描
- 用户未显式引用 @skill-discovery

### ✅ 触发条件
必须同时满足：
1. 用户明确说出"使用 skill-discovery"或"启动 skill-discovery"，或显式引用 @skill-discovery
2. 用户提供了明确需求或上下文

---

## 问题

在使用 agent 的过程中，我们积累了大量隐性知识和工作模式：

1. **反复纠正同一个行为**——每次都教，但从不沉淀
2. **同类修复反复出现**——同样的 bug 修了又修
3. **工作模式被遗忘**——某个流程跑通了，下次又从头摸索
4. **Anti-pattern 反复踩**——文档里写了但 agent 不看

**如果不主动提炼，这些能力就会流失。** 这个 skill 就是让"从经验中提炼能力"这件事本身变得可重复、系统化。

---

## 核心原则

```
主扫描(agent对话) → 模式浮现 → 验证补充(git/归档) → 覆盖度检查 → 用户判断 → 草案生成
```

每个阶段的设计理念：
- **主扫描**：agent 对话是用户与 agent 的真实互动，最能反映用户的工作方式和期望
- **模式浮现**：不预设类型（预防型、护栏型等），让模式从数据中自然涌现
- **验证补充**：git 和归档用于验证模式是否对应实际问题，而非发现模式
- **用户判断**：所有模式列出，由用户决定价值，不预设优先级
- **草案生成**：基于用户确认的模式，生成可用的 SKILL.md

---

## 执行流程

### Phase 1: 信号收集

**设计理念**：agent 对话是主扫描源，git 和归档是验证补充。

---

#### 1a. Agent 对话扫描（主扫描）

**扫描目标**：
- **Claude Code**: `~/.claude/projects/{当前项目路径}/*.jsonl`
- **Cursor**: `~/.cursor/projects/{当前项目路径}/agent-transcripts/*.jsonl`
- **Codex**: `~/.codex/history.jsonl`（单文件，不按项目分目录；需过滤当前项目相关的会话）
- 范围：最近 30 天的文件/记录

**执行步骤**：

```
Step 1: 定位 transcripts 目录
   - 根据当前工作目录推断项目路径
   - Claude Code: ~/.claude/projects/{project-name}/*.jsonl
   - Cursor: ~/.cursor/projects/{project-name}/agent-transcripts/*.jsonl
   - Codex: ~/.codex/history.jsonl（单文件；需按 cwd 字段过滤当前项目）
   - 筛选 mtime -30（最近30天）

Step 2: 提取用户消息
   - 读取 JSONL 文件，提取 type="user" 且 message.role="user" 的消息
   - 过滤掉：tool_result、<command-name>、<local-command>（非用户真实表达）
   - 截取内容前 300 字符（避免过长）

   **备注**：上述过滤逻辑基于 Claude Code 的 JSONL 格式。
   Cursor 和 Codex 的格式不同（Cursor 无 tool_result 字段，Codex 用 SQLite 而非 JSONL），
   需根据实际格式调整提取逻辑。

Step 3: 统计模式
   - 完全重复消息：统计出现 ≥3 次的完全相同消息
   - 关键词频率：统计用户消息中高频关键词（改、检查、问题、上下游、职责等）
   - 关键词组合聚类：相同关键词组合视为一类模式

Step 4: 输出模式清单
   - 每个模式包含：名称（从内容自然生成）、出现次数、典型片段
   - 不分类、不排序、不预设优先级
   - 所有模式列出，由用户判断价值
```

**模式示例**（实际扫描可能发现）：
```
模式: "全面检查改动上下游"
次数: 55 次
典型片段: "全面检查改动，仔细查看所有改动的上下游（调用者、被调用者），放在整个项目中考虑..."

模式: "规划要求"
次数: 52 次
典型片段: "规划一下"

模式: "卡住提醒"
次数: 17 次
典型片段: "你卡住了吗"
```

---

#### 1b. Git 提交验证（补充）

**定位**：验证 agent 对话发现的模式是否对应实际问题。

**执行步骤**：
```bash
# 检查是否有对应问题的反复修复
git log --all --oneline --grep="fix" --grep="修复" --grep="bug" -i -50
```

**验证逻辑**：
- 如果 git 中有对应问题的反复修复 → 模式有价值（用户教 agent 是因为 agent 真的会犯错）
- 如果 git 中从未出现对应问题 → 可能是用户偏好，需用户判断

---

#### 1c. 反馈归档验证（补充）

**定位**：验证 agent 对话发现的模式是否对应反复讨论的问题。

**扫描路径**：
- `docs/feedback/archive/`
- 用户指定路径

**验证逻辑**：
- 如果归档中有对应问题的反复讨论 → 模式有价值
- 如果归档中没有 → 可能是用户偏好，需用户判断

---

### Phase 2: 模式整理

将 Phase 1a 发现的模式整理为清晰清单：

**整理原则**：
- 不预设类型（预防型、护栏型等）
- 不预设优先级（高、中、低）
- 让模式名称从内容自然生成
- 所有模式列出，由用户判断

**输出格式**：
```
--- 发现模式 ---
模式: {从内容生成的名称}
次数: {出现次数}
验证: git中有/无对应修复，归档中有/无对应讨论
典型: {典型片段前100字}

--- 发现模式 ---
...
```

---

### Phase 3: 用户判断

将整理后的模式清单展示给用户，由用户判断价值：

**判断提示格式**：
```
共发现 X 个模式

--- 模式 1 ---
名称: {模式名称}
次数: {出现次数}
验证: {git/归档验证结果}
典型: {典型片段}

--- 模式 2 ---
...

请判断以上模式：
- 输入序号可查看单个模式详情
- 输入"接受 X"确认某个模式为 skill 候选（如"接受 1"）
- 输入"接受全部"确认所有模式为 skill 候选
- 输入"取消"中止，不生成草案
```

**用户判断维度**（不预设）：
- 是否对应实际问题？
- 是否有提炼价值？
- 是否值得固化成 skill？
- 由用户自行决定，agent 不预设答案

---

### Phase 4: 覆盖度检查

在确定候选模式后，检查现有 skill 是否已覆盖，决定行动方向。

#### 4a. 扫描现有 skill

扫描目标：
- **项目级**：`.claude/skills/`、`.cursor/rules/`、`AGENTS.md`
- **全局级**：`~/.claude/skills/`、`~/.cursor/skills/`

#### 4b. 判断覆盖情况

对每个候选模式，判断与现有 skill 的关系：

| 覆盖情况 | 判断标准 | 行动 |
|---------|---------|------|
| **完全覆盖** | 候选模式与现有 skill 功能完全一致 | 记录"已存在 skill: X"，不生成草案 |
| **部分覆盖** | 现有 skill 解决了部分问题，但仍有未覆盖的功能 | 建议修改现有 skill，标注需扩展的功能 |
| **无覆盖** | 无相关 skill 或完全不匹配 | 生成新 skill 草案 |

#### 4c. 输出覆盖度报告

```
--- 覆盖度检查 ---
候选: {skill-name}
覆盖情况: {完全覆盖/部分覆盖/无覆盖}
相关 skill: {skill 名称 或 "无"}
建议行动: {不生成草案/修改现有 skill/新建 skill}
```

---

### Phase 5: 草案生成策略

首先检查 my-create-skill 是否可用：

扫描 skill 目录：
- `.claude/skills/my-create-skill/SKILL.md`
- `.cursor/skills/my-create-skill/SKILL.md`

**如果可用**：
- 输出候选清单（名称、触发条件、优先级）
- 建议用户：`使用 my-create-skill 创建 {skill-name}`
- 本 skill 不生成完整草案，my-create-skill 会处理格式规范

**如果不可用**：
- 本 skill 生成简化草案框架（不含完整触发约束段落）
- 提醒用户：草案格式可能不完整，建议后续校验

简化草案格式：

```yaml
---
name: {skill-name}
description: {1句话描述，说明用途}
---

# {Skill 标题}

## 问题
{这个 skill 解决什么问题，用真实案例说明}

## 核心原则
{1-3 条核心原则}

## 执行流程
{步骤化的操作指南}
```

**草案质量要求**：
- MUST 包含真实案例（从扫描数据中提取）
- MUST 包含可执行的步骤
- NEVER 编造未扫描到的模式
- NEVER 生成空模板或占位符

---

### Phase 6: 输出与交付

生成 **能力发现报告**（文本形式）：

```
=== 能力发现报告 ===

扫描范围：agent transcripts（30天）、git（验证）、归档（验证）
发现模式：X 个

--- 模式清单 ---
1. {模式名称}
   次数: X 次
   验证: git有/无对应修复，归档有/无对应讨论
   典型: {片段前100字}

2. ...

--- 用户确认 ---
{列出用户确认的 skill 候选}

--- 建议行动 ---
my-create-skill 可用，建议执行：
  使用 my-create-skill 创建 {skill-name}
```

**输出选项**：
- 默认输出文本报告（对话中展示）
- 用户明确要求时可写入文件：
  - 报告：用户指定路径
  - 草案：`.claude/skills/{skill-name}/SKILL.md`（仅当 my-create-skill 不可用时）

---

### Phase 7: 用户确认与草案生成

用户确认后，生成草案：

**确认后的输出格式**：
```
--- 确认的 skill 候选 ---
候选 1: {skill-name}
模式来源: {模式名称}
触发条件: {从模式推断}
典型片段: {用于草案的问题说明}

--- 草案预览 ---
{简化草案内容}
```

**注意**：
- 此阶段只输出文本草案，不写入文件
- 用户需显式说"写到 XXX"才执行写入

---

## 硬约束

- **MUST 只读扫描** — 扫描阶段不修改任何项目数据
- **MUST 输出文本** — 报告和草案以对话文本形式输出，不默认写文件
- **MUST 用户指定写入路径** — 只有用户明确说"写到 XXX"时才写入
- **MUST 遵循格式** — 草案与用户已有全局 skill 格式一致
- **NEVER 自动写入文件** — 不写入 skill 目录、不写入临时文件、不写入任何位置
- **NEVER 编造模式** — 不凭空想象可能的 skill
- **NEVER 自动注册** — skill 的创建、注册、提交全部由用户手动完成
- **NEVER 扫描敏感数据** — 跳过 credentials、tokens、.env 等文件
- **NEVER 在扫描失败时静默跳过** — 明确报告哪些数据源不可用
