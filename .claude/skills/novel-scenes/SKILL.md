# 任务

将小说按**事件单元**拆分，为每个事件生成多维标签。支持多线索并行叙事。

| 调用方式 | 行为 |
|----------|------|
| `/novel-scenes {id} 1-30` | 手动指定章节范围，处理该范围内所有线索的事件 |
| `/novel-scenes {id}` | 默认处理前 30 章 |
| `/novel-scenes {id} all` | **三阶段处理**：扫描线索 → 识别事件 → 分批处理 |
| `/novel-scenes {id} scan` | 仅执行扫描：输出线索清单和事件边界（不生成文件） |

## 事件单元切分规则

### 核心概念

**事件单元 = 一个完整的戏剧动作（有起点、发展、结局）**

事件单元的特征：
- **线索归属**：每个事件属于一个叙事线索（主线/感情线/支线）
- **跨章聚合**：一个事件可能跨越 5-60 章
- **并行叙事**：同一章节可被多个线索的事件覆盖
- **不连续章节**：事件涉及的章节可以不连续（如 [1, 3, 5, 7]）

切分依据：
1. 角色状态变化（绝望→希望、敌对→合作）
2. 目标变化（求生→破案、复仇→守护）
3. 关系变化（陌生人→盟友、信任→背叛）
4. 明确的结局（成功/失败/悬念）

### 线索类型

| 线索类型 | thread 值示例 | 说明 |
|----------|---------------|------|
| 主线 | main | 主角核心叙事线 |
| 感情线 | romance_怀庆, romance_临安 | 特定角色的感情发展 |
| 支线 | subplot_云鹿, subplot_打更人 | 重要支线剧情 |
| 群像线 | cast_许新年, cast_魏渊 | 其他角色的个人故事 |
| 世界观线 | world_politics | 世界层面的叙事 |

**命名规则**：`{类型}_{区分标识}`，区分标识优先用角色名或地点名。

### 切分判断标准

**识别事件边界（何时切分）：**

1. **事件转折点**
   - 角色状态改变（从绝望到希望、从敌对到合作）
   - 目标变化（从"求生"到"破案"）
   - 关系变化（陌生人→盟友，信任→背叛）

2. **事件结局**
   - 有明确的结果（成功/失败/悬念）
   - 角色获得新信息或新能力
   - 情节达成阶段性终点

3. **线索切换**
   - 不同线索的事件在同一章交替出现时，分别归属各自线索
   - 同一线索的事件即使跨章也保持连贯

**不切分的情况：**

- 同一事件中的对话轮转
- 同一事件中的地点切换
- 同一线索内的时间流逝

## 钩子判定约束

钩子是事件与事件的关联点（铆合点），使孤立事件形成有机叙事。

### 钩子的本质

**钩子 = 跨事件的铆合点**

钩子是事件与事件之间的关联纽带，使孤立事件形成有机叙事。关键特征：
- 延迟足够长（至少跨事件或跨 10 章以上）
- 埋设内容与回收内容之间存在叙事关联
- 不在同一事件内揭示（否则是紧凑叙事，不是跨事件钩子）

### 钩子的四种铆合形式

钩子的铆合形式不限于反转，包括：

| 铆合形式 | 定义 | 示例 |
|----------|------|------|
| **反转铆合** | 第一次理解为 A，揭示后发现是 B | 热心路人 → 其实是反派卧底 |
| **悬念铆合** | 埋下疑问，驱动读者等待答案 | 章末"门外传来不可能出现的声音" → 下章揭示是谁 |
| **期待铆合** | 建立期待，让读者等待某个事件兑现 | 主角发誓要报仇 → 后面报仇成功 |
| **因果铆合** | 前事件的元素成为后事件的关键 | 第3章买的石头 → 第7章成为破局关键道具 |

**注意**：因果铆合不一定有反转。石头只是成为关键道具，不需要"原来石头是XX"的反转。铆合的核心是**建立关联**，而非制造反转

### 不是钩子的判定规则（排除规则）

以下情况**不是跨事件钩子**：

| 排除类型 | 定义 | 判定标准 |
|----------|------|----------|
| **单纯重复** | 同一物品/人物再次出现，没有建立新的叙事关联 | 埋设→回收之间没有新的意义（只是"又出现了"） |
| **自然延续** | 情节连贯性需要的过渡 | 没有铆合作用（埋设不直接影响回收事件走向） |
| **强调描写** | 作者明确用笔墨强调的细节 | 读者已注意到，不满足钩子的"延迟效果" |
| **间隔过短** | 埋设和回收在同一事件内 | 是事件内部紧凑叙事，不满足"跨事件"要求 |
| **无法验证** | 无法提供具体文本行号作为证据 | 可能是 AI 推断，置信度降为 low 或排除 |

### 钩子类型与置信度

| 类型 | 定义 | 置信度默认值 | 捕捉时机 |
|------|------|--------------|----------|
| **道具钩子** | 物品出场 → 跨事件发挥新作用 | high | 事件拆解时 |
| **悬念钩子** | 章末明确留下疑问 → 后期解答 | high | 事件拆解时 |
| **人物钩子** | 人物表面身份 → 后期身份/立场变化 | medium | refine 验证 |
| **信息钩子** | 闲笔细节 → 关键推理依据 | low | refine 验证 |
| **情感钩子** | 角色微妙反应 → 后期揭示原因 | low（需文本证据） | refine 验证 |

**置信度调整规则**：
- 有明确文本行号引用 → 可保持默认值
- 无行号引用 → 降一级（high→medium, medium→low, low→排除）
- 触犯排除规则 → 直接排除

### 钩子判定验证流程

识别疑似钩子时，执行以下验证：

```
验证项1: 铆合形式
  - 属于哪种铆合形式？（反转/悬念/期待/因果）
  - 填写埋设→回收的关联说明
  
验证项2: 延迟长度
  - 延迟长度: [埋设到回收的章数]
  - 是否跨事件？是/否
  - 同一事件内的延迟不满足"跨事件"要求
  
验证项3: 文本证据
  - 埋设位置: 第N章，行号范围 [X-Y]
  - 回收位置: 第M章，行号范围 [A-B]（已知时填写）
  - 无行号引用 → 置信度降级
```

### 钩子扫描时机

**分层处理策略**：

| 阶段 | 负责内容 | 原因 |
|------|----------|------|
| **事件拆解（novel-scenes）** | 章末悬念 + 道具钩子 | 最显性，可从单个事件识别 |
| **精调验证（refine）** | 人物钩子 + 信息钩子 + 情感钩子 | 需要跨事件视角，隐性难识别 |

事件拆解时：
- 捕捉本事件内的章末悬念
- 捕捉本事件涉及的道具（可能成为跨事件钩子）
- 标记置信度，等待 refine 验证

精调时：
- 扫描全部事件，识别人物/信息/情感钩子
- 验证事件拆解时标记的疑似钩子
- 建立完整的钩子铆合链

## 钩子扫描模板

事件拆解时，执行以下扫描流程捕捉显性钩子。

### A. 章末悬念扫描

对本事件涉及的每章结尾，检查是否存在章末悬念。

**扫描步骤**：
1. 定位每章末尾段落（最后 5-10 行）
2. 检查是否有以下特征：
   - 关键动作暂停（"他推开了门——"）
   - 悬念人物出现（"门外传来不可能出现的声音"）
   - 关键信息揭示但留疑问（"真相只有一个，但——"）
   - 角色进入即时危险（"身后传来脚步声"）

**输出格式**：

```yaml
# 写入事件的 hooks.chapter_end 字段
chapter_end:
  - chapter: 5
    type: 悬念人物          # 悬念人物/动作暂停/信息揭示/即时危险
    hook_text: '门外传来不可能出现的声音'
    lines: [450, 455]       # 行号范围
    resolved_in: unknown    # 回收事件（未知时填 unknown）
```

**注意事项**：
- 章末悬念是最显性钩子，置信度默认 high
- 同一事件可能有多个章末悬念（不同章节）
- 回收事件未知时填 `unknown`，等待 refine 验证

### B. 跨事件道具扫描

提取本事件涉及的物品，识别可能成为跨事件钩子的道具。

**扫描步骤**：
1. 扫描本事件涉及的章节正文
2. 提取所有明确命名的物品
3. 检查是否有以下特征：
   - 物品首次出场时描述平淡（读者不特别注意）
   - 物品在本事件未发挥关键作用（仅出场/携带/购买）
   - 物品可能在后续事件有新作用（需要推断）

**判定规则**：

| 情况 | 是否钩子 | 说明 |
|------|----------|------|
| 物品本事件内发挥关键作用 | 否 | 不满足"跨事件"要求 |
| 物品只是背景描写（如"桌上放着一杯茶"） | 否 | 不满足"铆合作用"要求 |
| 物品首次出场但本事件未用，后续可能用 | 疑似钩子 | 置信度 medium，待 refine 验证 |
| 物品已明确在后续事件发挥新作用 | 是 | 置信度 high（需要文本证据） |

**输出格式**：

```yaml
# 写入事件的 hooks.items_crossing 字段
items_crossing:
  - item: 神秘石头
    planted_chapter: 3
    planted_lines: [120, 125]
    planted_event: ev_main_001           # 本事件ID
    role_in_event: 背景道具              # 本事件中的角色（关键道具/背景道具/携带物品）
    expected_harvest: unknown            # 预计回收事件（未知时填 unknown）
    crossing_type: 因果铆合              # 因果铆合/反转铆合（已知时填写）
    confidence: medium                   # high/medium/low
```

### C. 跨事件人物关联扫描

提取本事件涉及的人物，标记可能存在身份/立场变化的人物。

**扫描步骤**：
1. 提取本事件涉及的所有人物
2. 检查是否有以下特征：
   - 人物出场时有微妙描写（表情复杂、沉默、异样眼神）
   - 人物在本事件行为与预期有差异
   - 人物可能有隐藏动机（需要推断）

**判定规则**：

| 情况 | 是否标记 | 说明 |
|------|----------|------|
| 人物行为完全符合预期 | 否 | 无钩子迹象 |
| 人物有微妙描写但未明确异常 | 疑似钩子 | 置信度 medium，待 refine 验证 |
| 人物已明确揭示身份/立场变化 | 是 | 置信度 high（需要文本证据） |

**注意**：人物钩子往往是隐性，事件拆解时只标记疑似，由 refine 验证。

**输出格式**：

```yaml
# 写入事件的 hooks.character_crossing 字段
character_crossing:
  - character: 热心路人
    first_appearance: 3
    first_event: ev_main_001
    behavior_hint: 出场时眼神有微妙异样    # 行为异常描述（可选）
    expected_role_change: unknown         # 预计变化类型（未知时填 unknown）
    confidence: medium                    # 默认 medium，refine 验证后可调整
```

### D. 信息钩子标记（低置信度）

提取对话和背景描写中的细节，标记可能成为关键线索的闲笔。

**扫描步骤**：
1. 扫描对话中的"闲聊"内容（与主线无关的对话）
2. 扫描背景描写中的细节（环境、物品、天气等）
3. 标记可能有钩子价值的细节

**判定规则**：
- 信息钩子最难识别，事件拆解时置信度默认 low
- 不在本事件验证，只标记疑似，由 refine 验证

**输出格式**：

```yaml
# 写入事件的 hooks.info_hint 字段（可选）
info_hint:
  - detail: '城东的井水最近变苦了'
    chapter: 12
    lines: [280, 285]
    context: 闲聊对话                     # 出现场景（闲聊对话/背景描写/人物独白）
    confidence: low                       # 默认 low
```

### E. 钩子扫描汇总输出

完成扫描后，输出汇总报告：

```
🪝 钩子扫描结果（事件 {event_id}）

章末悬念：{N}个
  第X章: [悬念人物] "门外传来..." → 回收事件 unknown
  第Y章: [动作暂停] "他推开了门——" → 回收事件 unknown

跨事件道具：{M}个
  道具A: 第Z章出场 → 预计回收 unknown → 置信度 medium
  道具B: 第W章出场 → 预计回收 unknown → 置信度 low

人物关联标记：{K}个（待 refine 验证）
  人物A: 第P章出场 → 行为有微妙异常 → 置信度 medium

信息钩子标记：{L}个（待 refine 验证）
  细节X: 第Q章闲聊 → 置信度 low

总计：{N+M+K+L}个钩子标记
置信度分布：high {H}个, medium {M}个, low {L}个
```

## Schema

输出遵循 `docs/schemas/event-unit.schema.yaml`。

**关键格式约束：**
- 事件ID格式：`ev{线索}_{序号}`
  - 主线事件：`ev_main_001`, `ev_main_002`
  - 感情线事件：`ev_romance_怀庆_001`
  - 支线事件：`ev_subplot_云鹿_001`
- `chapters` 可为不连续列表：`[1, 3, 5, 7]`
- 同一章可被多个事件覆盖（不同 thread）

## 三阶段处理流程

### 为什么需要三阶段？

**问题**：
1. 事件可能跨越数十章，按固定章数切分会打散事件
2. 同一章节可能涉及多个并行线索（主线+感情线）
3. 单线索视角下，事件涉及的章节可能不连续

**解决方案**：
1. **第一阶段：线索识别** — 识别小说中的并行叙事线索
2. **第二阶段：事件边界扫描** — 每个线索内识别事件边界
3. **第三阶段：按事件分批处理** — 根据事件范围分配批次

---

### 第一阶段：线索识别

#### 1a. 读取 outline 获取线索信息

读取 `outline.yaml`，提取：
- `timelines` 列表（如有）获取多线叙事结构
- 主要角色列表，推断感情线/群像线
- 势力/组织信息，推断世界观线

#### 1b. 扫描章节标题识别线索

扫描 `source.txt` 的章节标题，识别：
- 主线关键词（破案、战斗、修炼、升级）
- 感情线关键词（告白、约会、误会、复合）
- 支线关键词（特定角色名、地点名）

#### 1c. 输出线索清单

```yaml
# threads_manifest.yaml
threads:
  - id: main
    description: 许七安主线叙事
    estimated_chapters: "大部分章节"
    priority: 1
    
  - id: romance_怀庆
    description: 怀庆公主感情线
    estimated_chapters: [50, 120, 200, 350, ...]  # 预估涉及章节
    priority: 2
    
  - id: romance_临安
    description: 临安公主感情线
    estimated_chapters: [80, 150, 280, ...]
    priority: 2
    
  - id: subplot_魏渊
    description: 魏渊个人线
    estimated_chapters: [100, 250, 400]
    priority: 3
```

---

### 第二阶段：事件边界扫描

#### 2a. 按线索扫描事件边界

对每个线索单独扫描，识别该线索内的事件边界。

**主线扫描**：读取全部章节标题，识别主线事件
**支线扫描**：只读取涉及该支线的章节（基于第一阶段预估）

#### 2b. 生成事件边界清单

```yaml
# events_manifest_draft.yaml
thread: main
events:
  - id: ev_main_001
    title: 税银案破案自救
    chapters: [1, 2, 3, 4, 5]
    event_type: 推理破案
    outcome: 成功自救
    confidence: high
    
  - id: ev_main_002
    title: 观书台修行
    chapters: [6, 8, 10, 12]  # 不连续
    event_type: 修炼突破
    outcome: 能力觉醒
    confidence: medium

thread: romance_怀庆
events:
  - id: ev_romance_怀庆_001
    title: 初遇怀庆
    chapters: [50]
    event_type: 告别重逢
    outcome: 关系确立
    confidence: high
    
  - id: ev_romance_怀庆_002
    title: 怀庆暗助
    chapters: [120, 125, 130]  # 不连续
    event_type: 暗中相助
    outcome: 感情升温
    confidence: medium
```

#### 2c. 输出扫描结果

```
📊 线索与事件边界扫描完成

📚 素材：{name}
📖 总章节：849章

识别到的线索：
  main: 主线（预计覆盖 849 章）
  romance_怀庆: 怀庆感情线（预计覆盖约 50 章）
  romance_临安: 临安感情线（预计覆盖约 40 章）
  subplot_魏渊: 魵渊个人线（预计覆盖约 30 章）

主线事件：
  ev_main_001: 税银案破案自救 [第1-5章]
  ev_main_002: 观书台修行 [第6,8,10,12章]
  ...

感情线（怀庆）事件：
  ev_romance_怀庆_001: 初遇怀庆 [第50章]
  ...

预估事件数：主线 {N}个，感情线 {M}个，总计 {N+M}个

建议使用 /novel-scenes {id} all 开始处理
```

---

### 第三阶段：按事件分批处理

#### 3a. 线索优先级排序

按优先级处理线索：
1. 先处理主线（priority=1）
2. 再处理感情线（priority=2）
3. 最后处理支线（priority=3）

#### 3b. 事件分配策略

| 事件规模 | 处理方式 |
|----------|----------|
| ≤10章 | 单批处理 |
| 10-30章 | 单批处理（增大批次容量） |
| 30-50章 | 单批处理（需要大 context window） |
| >50章 | 多批处理，保持同一事件ID |

#### 3c. 不连续章节的处理

当事件 `chapters` 不连续时：
```
处理策略：
1. 第一批：处理首批章节（如 [1, 3, 5]）
   - 读取这些章节的相关段落
   - 生成事件骨架
   
2. 中间批：处理后续章节（如 [7, 9, 11]）
   - 增量补充信息
   - 更新 event_arc
   
3. 最后一批：处理末尾章节
   - 确定 outcome
   - 完成事件单元
```

#### 3d. 为每个事件生成标签

参照 `docs/schemas/event-unit.schema.yaml`，填写：

**A. 事件层**
- `id`, `thread`, `event_type`, `event_stage`, `outcome`

**B. 人物层**
- `characters`, `primary_character`, `relationship`, `interaction`

**C. 情感层**
- `emotion_arc`, `tension_peak`, `tension_curve`

**D. 结构层**
- `plot_stage`, `plot_function`, `pacing`

**E. 技法层**
- `technique`, `dialogue_type`, `pov`

**F. 物理层**
- `setting`, `scale`, `time_weather`

#### 3e. 写入文件

```
events/

  ev_main_002.yaml
  ev_romance_怀庆_001.yaml # 怀庆感情线事件
  ev_subplot_魏渊_001.yaml # 魏渊支线事件
```

```bash

python scripts/core/quality_audit.py {material_id} --event {event_id}
```

### 第四阶段：覆盖检查

**多线索覆盖规则**：同一章节可被多个事件覆盖。

检查逻辑：
1. 主线应覆盖所有章节
2. 支线只覆盖其涉及的章节（不要求全覆盖）
3. 检查是否有章节未被任何主线事件覆盖

```
覆盖检查结果：
  第1章: ev_main_001 ✓
  第50章: ev_main_010 + ev_romance_怀庆_001 ✓（多事件覆盖）
  第120章: ev_main_025 + ev_romance_怀庆_002 ✓
  
  缺失章节：无
  
  或：
  缺失章节：第543章（主线未覆盖）
  → 自动补处理
```

## 上下文控制策略

| 策略 | 说明 |
|------|------|
| 线索扫描只读标题 | 不读正文，控制 token |
| 支线只处理涉及章节 | 不处理无关章节 |
| 不连续章节分批处理 | 每批只传递前批压缩摘要 |
| 线索间独立处理 | 不同线索的事件不互相传递上下文 |

## YAML 书写安全

| 规则 | 正确 | 错误 |
|------|------|------|
| 含中文引号的值用单引号包裹 | `note: '许七安说"活下去"'` | `note: 许七安说"活下去"` |
| summary 超过一行用 `>-` | `summary: >-` 换行写内容 | 直接写超长单行 |
| 数值字段不加引号 | `tension_peak: 5` | `tension_peak: "5"` |

## 注意事项

- 每个事件必须指定 `thread`（线索归属）
- 同一章节可被多个线索的事件覆盖
- 事件章节可以不连续，按实际涉及章节填写
- 主线应覆盖全部章节，支线只覆盖涉及章节
- summary 概括事件全过程，不截断首句
- 标签值必须从字典选取，缺值时用 `/tag-add` 补充

## 质量红线

| # | 检查项 | 判定标准 |
|---|--------|----------|
| 1 | 主线覆盖缺失 | 主线存在未覆盖的章节 |
| 2 | 标签雷同 | 同一线索内 ≥50% 事件 `event_type` 相同 |
| 3 | summary 雷同 | ≥2 个事件 summary 仅章节号不同 |
| 4 | title 无语义 | 存在 `title: 事件N` 或纯编号 |
| 5 | YAML 解析失败 | 任何事件文件 `yaml.safe_load` 报错 |
| 6 | thread 缺失 | 事件未指定线索归属 |
| 7 | 钩子无文本证据 | 钩子标记缺少行号引用（置信度强制降级） |
| 8 | 钩子触犯排除规则 | 钩子标记触犯"不是钩子"的排除规则（强制删除） |
| 9 | 钩子置信度不合理 | 钩子置信度与类型不匹配（如人物钩子标 high） |
| 10 | 钩子铆合形式缺失 | 钩子未标注铆合形式（反转/悬念/期待/因果） |

### 钩子质量检查流程

对已标记的钩子执行以下检查：

**检查清单**：

```
1. 证据锚定检查
   - 是否有埋设位置的行号引用？
   - 回收位置已知时是否有行号引用？
   - 无行号 → 置信度强制降级
   
2. 排除规则检查
   - 是否触犯"不是钩子"的排除规则？
   - 触犯 → 强制删除该钩子标记
   
3. 置信度合理性检查
   - 道具钩子 → 默认 high，无行号时降为 medium
   - 悬念钩子 → 默认 high，无行号时降为 medium
   - 人物钩子 → 默认 medium，无行号时降为 low
   - 信息钩子 → 默认 low，无行号时删除
   - 情感钩子 → 需有明确文本证据，否则删除
   
4. 铆合形式检查
   - 是否标注铆合形式？
   - 未标注 → 补充标注（根据内容推断）
   
5. 跨事件验证（refine阶段）
   - 验证钩子是否真正跨事件
   - 验证钩子是否在后续事件回收
   - 未回收 → 标记为"待回收"，记录在钩子清单
```

**检查输出格式**：

```
🪝 钩子质量检查结果（事件 {event_id}）

检查项：{total}个钩子标记

通过：{pass}个
  - hook_001: ✓ 证据锚定+置信度合理
  - hook_002: ✓ 排除规则未触犯+铆合形式已标注

需修正：{fix}个
  - hook_003: ⚠ 缺行号引用 → 置信度降为 low
  - hook_004: ⚠ 铆合形式缺失 → 补充"因果铆合"

删除：{delete}个
  - hook_005: ✗ 触犯"单纯重复"排除规则 → 强制删除
  - hook_006: ✗ 情感钩子无文本证据 → 强制删除

质量红线触发：{trigger}项（红线7-10）
```

### 钩子红线触发后的处理

| 红线 | 处理方式 |
|------|----------|
| 红线7（无文本证据） | 置信度强制降级，不删除标记 |
| 红线8（触犯排除规则） | **强制删除**该钩子标记 |
| 红线9（置信度不合理） | 调整置信度为合理值 |
| 红线10（铆合形式缺失） | 补充铆合形式标注 |

**重要**：红线8触犯时必须删除，不能保留低置信度版本。
