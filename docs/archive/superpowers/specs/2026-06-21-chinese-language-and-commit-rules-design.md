# 中文沟通与 Git 提交规则设计

## 目标

为当前用户的所有 Codex 项目建立统一中文沟通规范，并在 Novel Material 项目内再次声明该规范，确保全局默认与项目约束同时生效。

## 规则载体

- 全局规则写入 `~/.codex/AGENTS.md`，对当前用户启动的所有 Codex 项目生效。
- 项目规则写入仓库根目录 `AGENTS.md`，便于项目独立使用、复制或交给其他 Agent 时仍保留约束。
- 不修改 `~/.codex/config.toml`，因为语言与提交规范属于持久指导，而不是运行配置。

## 中文沟通规则

- 所有面向用户的回复、计划、任务说明、进度更新、分析结论、决策依据、错误说明和新编写的说明性文档默认使用简体中文。
- 内部隐藏推理不会展示；所有可见的推理摘要和判断依据使用中文。
- 命令、代码、路径、配置键、字段名、API 名称、模型名称和无法准确翻译的专有名词可以保留原文。
- 用户明确要求其他语言时，以用户当前请求为准。

## Git 提交规则

提交标题沿用项目既有 Conventional Commits 习惯：

```text
<type>(<scope>): <中文摘要>
```

- `type` 使用 `feat`、`fix`、`refactor`、`docs`、`test`、`chore`、`skills` 等既有英文类型。
- `scope` 使用实际模块名，可省略。
- 除类型、作用域、代码标识、文件名、命令和技术专名外，标题与正文必须使用中文。
- 每次提交必须包含正文，禁止只有标题。
- 正文中的“主要改动”和“验证结果”必须存在；“背景与目的”“影响范围”按实际情况填写。
- 主要改动使用具体内容点说明，不写“更新代码”“修复问题”等无法审计的笼统描述。
- 验证结果必须记录执行的测试或检查及结果；未运行测试时明确说明原因。

推荐格式：

```text
<type>(<scope>): <中文摘要>

背景与目的：
- <修改原因，可按需省略本节>

主要改动：
- <具体改动点>
- <具体改动点>

影响范围：
- <兼容性、数据或行为影响，可按需省略本节>

验证结果：
- <执行的命令或检查>
- <通过结果，或未执行测试的原因>
```

## 生效与验证

- 修改规则文件后，当前对话立即按新规则执行。
- Codex 会在新会话启动时重新加载全局和项目 `AGENTS.md`；完整验证需要新建会话。
- 检查项目文件与全局文件均包含中文沟通、例外范围和 Git 提交正文要求。
- 本次规则修改不触碰 `config/providers.yaml`、`docs/feedback.md` 等用户现有工作区改动。
