# 素材分类功能重构问题分析

## 背景

素材分类功能（classify）已实现基础版本，但存在三个核心问题：
1. 标签体系与现有系统脱节
2. 章节取样不够科学
3. 信息提取利用率低

---

## 问题一：标签体系脱节

### 现状

| 维度 | classify 现状 | 系统标签体系 |
|------|---------------|-------------|
| Genre | 10 个一级（硬编码） | 20 个一级 + 80+ 二级 |
| Element | 无 | 200+ 个（爽点、设定、叙事） |
| Style | 无 | 85+ 个（风格、基调、节奏） |
| Character | 无 | 150+ 个（原型、身份、关系） |

**代码位置**：`material/classify_prompt.py:38-49`

```python
VALID_GENRES = {
    "玄幻", "修仙", "东方玄幻",  # 仅 18 个
    "奇幻", "西幻", "魔幻",
    ...
}
```

**系统标签位置**：
- `tags/load.py` - `load_tags_for_genre()` 动态加载
- `tags/validate.py` - `validate_tag()` 校验并归一化
- 数据库 `genre_domain_map` 表 - 题材与领域映射

### 影响

1. 分类结果无法用于后续 pipeline（入库时 genre 不匹配）
2. LLM 输出的 "官场"、"末世" 等被过滤为 "其他"
3. 无法按题材精准检索候选素材

---

## 问题二：章节取样不科学

### 现状

**代码位置**：`material/classify.py:40-98`

```python
def extract_first_three_chapters(file_path, max_chars=8000):
    # 只取前三章，最多 8000 字
```

### 问题分析

| 小说类型 | 总章数 | 前三章占比 | 0.5% 占比 | 问题 |
|---------|-------|-----------|----------|------|
| 短篇（50章） | 50 | 6% | 0.25章 | 前三章过度代表 |
| 中篇（200章） | 200 | 1.5% | 1章 | 前三章可能够 |
| 长篇（1000章） | 1000 | 0.3% | 5章 | 前三章信息不足 |
| 超长篇（3000章） | 3000 | 0.1% | 15章 | 严重信息缺失 |

**具体问题**：
1. 前三章可能是铺垫、介绍，核心设定未展开
2. 都市异能、悬疑类小说，前三章看不出 genre
3. 长篇小说的世界观在中后期才完整呈现

### 影响

- 分类置信度低（如《诡秘之主》仅靠前三章可能误判）
- genre_description 不够准确
- 无法捕捉小说后期的转折/进化

## 问题三：信息提取利用率低

### 现状

classify 调用一次 LLM（约 7000 tokens 输入 + 2500 输出），但只提取：

```json
{
  "genre": ["仙侠", "修仙"],
  "genre_description": "...",
  "confidence": 0.95
}
```

### 可提取但未提取的信息

| 信息类型 | 价值 | 后续用途 |
|---------|------|---------|
| 核心元素 | 识别爽点、设定 | 按爽点检索（如"重生"、"系统"） |
| 风格基调 | 识别叙事风格 | 按风格筛选（如"快节奏"、"热血"） |
| 主角原型 | 识别主角类型 | 人物模板匹配 |
| 热度预估 | 预测适合平台 | 平台适配建议 |
| 入库优先级 | 评估入库价值 | 智能入库排序 |
| 写作质量 | 评估文笔水平 | 筛选高质量素材 |

### 影响

- 每次 LLM 调用花费约 0.02-0.05 元，但只利用 30% 的分析能力
- 分类结果不够丰富，无法支持按需入库的精细化筛选
- 与后续 pipeline 的 tags 分析重复调用

## 统计数据

### 分散程度

| 检查项 | 结果 |
|--------|------|
| 现有 genre 定义分散 | `classify_prompt.py`（18个） vs `data/tag-system/03-genre.md`（80+） |
| 标签校验分散 | `classify.py` 自己校验 vs `tags/validate.py` 统一校验 |
| 样本提取函数 | 1 处（`extract_first_three_chapters`） |

### 调用关系

```
classify.py
    ├── classify_prompt.py（硬编码 genre）
    ├── infra/llm.py（调用 LLM）
    └── yaml_io.py（保存结果）
    
# 未调用：
# tags/load.py - 动态加载标签
# tags/validate.py - 校验归一化
```

---

## 重构目标

1. **标签对接**：使用系统标签体系，分类结果可直接入库
2. **科学取样**：约 0.5% 章节（最少 3 章，最多 30 章），分布式采样
3. **信息扩展**：一次调用提取 genre + elements + style + quality

---

## 验收标准

1. 分类结果 genre 与 `genre_domain_map` 匹配
2. 取样覆盖开头 + 中间 + 后期（可配置，最多 30 章）
3. 输出包含 genre + elements + style + quality 四个维度
4. 单次调用 token 消耗控制在 15000 以内（30 章 × 1500 字）
