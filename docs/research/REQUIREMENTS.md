# 项目需求文档

> 本文档记录 novel-material 项目的核心用户需求，所有架构设计和技术决策必须服务于这些需求。

## 一、项目定位

**novel-material 是一个小说写作参考检索库。**

它不是用来"学习规律"的训练系统，而是用来**提供检索参考**的素材库。

**类比**：建筑师不会从 100 栋建筑中"学习"如何设计，而是翻阅优秀案例图集，把多个案例的精华组合到自己的设计中。本项目就是那个"案例图集"。

---

## 二、用户的两个写作场景

### 场景 A：前期规划阶段

用户在动笔之前，需要构建小说的整体框架。

**需要辅助生成的内容：**
- 世界观设定
- 规则体系（力量体系、社会规则等）
- 大纲（全书结构）
- 细纲（幕/序列/节拍级结构）

**用户工作流：**
1. 用户与 Agent 讨论，明确自己想要什么样的世界观/大纲
2. Agent 从本项目中检索 N 条匹配度最高的参考
3. 用户 + Agent 将多条参考糅合，形成自己的设定

### 场景 B：章节写作阶段

用户在写具体章节时，需要参考同类章节的写法。

**需要辅助生成的内容：**
- 章纲（单章节的结构和推进方式）
- 人物描写（人物的出场、互动、弧光展现）
- 事件描写（特定类型事件的写法）

**用户工作流：**
1. 用户写到某个章节/人物/事件时，与 Agent 讨论需要什么参考
2. Agent 从本项目中检索 N 条匹配度最高的参考
3. 用户 + Agent 将多条参考糅合，写出自己的内容

## 三、6 个核心检索需求

### 需求 1：世界观检索

**触发时机**：场景 A — 前期规划

**用户意图**：找到类似类型/设定的世界观参考

**查询示例**：
- "找修仙类小说的力量体系设计，分 5 个等级的"
- "找有多个势力博弈的世界观"
- "找地理空间复杂（多大陆/多星球）的设定"

**返回内容**：力量体系结构 + 势力关系图 + 地理空间描述 + 设定亮点

### 需求 2：大纲检索

**触发时机**：场景 A — 前期规划

**用户意图**：找到类似类型/结构的大纲作为参考

**查询示例**：
- "找 3 本修仙小说的大纲，主角都是废柴逆袭"
- "找开局是困境模式的小说大纲"
- "找多线叙事的玄幻大纲"

**返回内容**：完整的大纲结构树（幕 → 序列 → 节拍）

### 需求 3：细纲检索

**触发时机**：场景 A — 前期规划

**用户意图**：找到类似情节推进模式的细纲

**查询示例**：
- "找悬疑类小说的中段推进模式"
- "找感情线从相识到相爱的节拍设计"
- "找高潮前的铺垫序列"

**返回内容**：序列级/节拍级的结构对比

### 需求 4：章纲检索

**触发时机**：场景 B — 章节写作

**用户意图**：找到同类型章节的写法参考

**查询示例**：
- "找 10 本小说第 1 章的开局写法"
- "找高潮章节的写法"
- "找转折点章节的写法"

**返回内容**：章节摘要 + 章节功能标签 + 结构信息

### 需求 5：人物检索

**触发时机**：场景 B — 人物写作

**用户意图**：找到同类人物的塑造参考

**查询示例**：
- "找修仙类小说的导师型人物写法"
- "找反派角色的出场方式"
- "找配角的人物弧光设计"

**返回内容**：人物小传 + 关键出场章节 + 互动模式

### 需求 6：事件检索

**触发时机**：场景 B — 事件写作

**用户意图**：找到同类事件的写法参考

**查询示例**：
- "找雨中告别的写法"
- "找主角初次突破的写法"
- "找团队内讧的写法"

**返回内容**：匹配的章节摘要 + 上下文信息

## 四、检索接口定义

### 4.1 检索请求格式

```yaml
query: "用户自然语言查询"              # 必填
search_type: chapter                  # 必填: world/outline/detail/chapter/character/event
filters:                              # 可选：结构化过滤
  genre: [修仙, 玄幻]
  chapter: 1
  chapter_function: [开局困境]
  archetype: 导师
limit: 10                             # 可选：返回数量，默认 5
include_embedding: true               # 可选：是否返回向量相似度
```

### 4.2 检索响应格式

```yaml
results:
  - material_id: nm_xxx
    novel_name: "《xxx》"
    match_score: 0.92                 # 综合匹配度
    match_reason:                     # 匹配原因说明
      - "genre 匹配: 修仙"
      - "向量相似度: 0.95"
      - "章节功能: 开局困境"
    # 以下为各类型专属数据
    chapter_summary: "..."            # 章纲检索时返回
    chapter_functions: [...]          # 章纲检索时返回
    outline_tree: {...}               # 大纲/细纲检索时返回
    character_profile: {...}          # 人物检索时返回
    worldbuilding_summary: {...}      # 世界观检索时返回
```

### 4.3 调用方式

```bash
# 脚本调用（推荐）
python scripts/core/search.py chapter --query "开局困境写法" --genre 修仙 --limit 10

# Agent 直接读取（小规模/无数据库时）
# 通过 YAML 索引文件读取
```

## 五、关键约束与原则

### 5.1 标注数据的定位

**标注数据是"检索参考"，不是"学习材料"。**

Agent 本身有能力：
- 从阅读过的海量文本中隐式学习模式
- 根据用户指令生成结构
- 参考具体样例调整输出

**不需要**从标注数据中"学习规律"来生成小说。标注数据的作用是让 Agent 能**检索到高质量参考样例**，然后糅合使用。

### 5.2 拆分悖论

**拆分越精细，边界越难定义；拆分越粗，规律提取越依赖理解。**

| 拆分层级 | 边界困难 | 规律提取困难 |
|----------|----------|--------------|
| 事件 | 戏剧动作模糊（不可控） | 需要理解事件角色 |
| 场景 | 焦点切换信号（半可控） | 需要理解场景关系 |
| 章节 | 无困难（完全可控） | 需要理解章节内容 |

**结论**：以章节为最小分析单元。边界完全可控，检索效果足够。

### 5.3 Agent 的核心能力

**Agent 自己有能力从参考中理解结构，不需要人为标注转折/高潮等结构角色。**

例如，Agent 看到：
- 第一章摘要：入狱 → 困境 → 自救思路 → 能理解"开局范式"
- 第 1-5 章摘要：破案过程 → 能理解"推进范式"
- 第 7 章摘要：家人团聚 + 配角自杀悬念 → 能理解"转折范式"

这些理解 Agent 可以自己完成，不需要人为标注。

### 5.4 糅合机制

用户说的"把 N 条匹配度最高的参考糅合在一起"，具体含义是：

1. **检索**：Agent 根据用户需求找到 N 条最匹配的参考
2. **展示**：将 N 条参考以结构化形式展示给用户和 Agent
3. **理解**：Agent 分析每条参考的特点（结构、节奏、技法等）
4. **糅合**：结合用户的个性化需求，将多条参考的精华整合
5. **生成**：产出用户自己的大纲/细纲/章纲/人物/事件

**本项目负责的是第 1-2 步**（检索 + 展示）。第 3-5 步由 Agent 在写作场景中完成。

### 5.5 物品/怪物/能力的定位

物品、怪物、能力等世界观元素**不单独设立检索场景**。

**原因**：
- 设定本身不可照搬（每部小说的世界观独立）
- 有价值的不是"物品长什么样"，而是"关键道具在故事里起什么作用"
- 后者已由**事件检索**覆盖（"关键道具登场"怎么写、"能力突破"怎么写）

**但需要在世界观检索中增强**：力量体系的结构化数据（等级、突破条件、代价）。

### 5.6 素材独立性原则

**每部小说的世界观、人物、事件相互独立，不跨书共享。**

检索时返回的是"参考样例"，不是"直接引用"。糅合后生成的内容属于用户自己的创作。

## 六、数据生命周期

### 6.1 入库流程

```
原文文件 → 格式清洗 → 章节切分 → 章级分析(LLM) → Embedding → 写入数据库
    ↓           ↓          ↓          ↓             ↓          ↓
source.txt  format_report  chapter_index  chapters.yaml  向量数据  PostgreSQL
```

**每个环节都有质量校验**：
- 格式清洗：章节名识别率、缺章检测
- 章级分析：摘要完整性、标签合法性
- Embedding：向量维度校验

### 6.2 更新机制

- **增量更新**：新小说入库后，自动加入全局索引，无需重建
- **全量重建**：Schema 变更时，从 YAML 源数据重建数据库
- **版本控制**：YAML 文件作为 Source of Truth，数据库是派生查询层

### 6.3 删除与清理

- 删除小说时同步清理数据库中所有关联记录
- 记录删除日志（`deletion_log.yaml`）

### 7.3 关键维度速查

**小说题材（genre）**：东方玄幻、异世大陆、修真文明、都市异能、架空历史、星际文明、古代言情、现代言情、快穿等

**核心元素（element）**：重生、穿越、系统、签到、无敌、废柴、种田、末世、直播、国运、无限流、诸天流、赘婿、神医、风水等

**人物原型（character archetype）**：英雄、导师、伙伴、反派、隐士、复仇者、守护者、野心家、腹黑、傲娇、病娇等

**叙事风格（prose style）**：华丽、朴素、冷叙述、诗化、写实、讽刺、幽默、古风、赛博朋克风等

**章节功能（chapter function）**：开局困境、悬念引入、人物亮相、线索发现、冲突升级、情节反转、终极对决、悬念揭晓、章末悬念等

## 九、成本预算

### 9.1 入库处理成本

| 环节 | 每章成本 | 1000本(50万章)成本 | 说明 |
|------|---------|-------------------|------|
| 章节切分 | ~0.01秒 | ~5000秒 | 纯脚本，机械操作 |
| 章级分析（LLM） | ~2秒 | ~166小时 | 可分批并行，10并发 ~17小时 |
| Embedding | ~0.1秒 | ~8小时 | 本地模型或API |
| 数据库写入 | ~0.01秒 | ~5000秒 | 批量插入 |

### 9.2 推理成本

| 环节 | 成本说明 |
|------|---------|
| 章级分析 LLM | 每章约 2K tokens 输入 + 500 tokens 输出 |
| Embedding | 每章 1 次 embedding 调用（约 0.5K tokens） |

### 9.3 存储成本

| 数据 | 估算 |
|------|------|
| 原文文本 | ~5GB（50万章 × 10KB/章） |
| 章级分析数据 | ~500MB |
| 向量数据（1536维 float32） | ~3GB |
| 数据库总大小 | ~10-15GB |

## 十、不做的内容

基于上述需求，以下方向**明确不做**：

| 不做内容 | 原因 |
|----------|------|
| 事件拆分（"戏剧动作"级别） | 边界不可控，投入产出比极低 |
| 场景拆分 | 切分本身不产生规律，用户反馈难提取 |
| 结构角色标注（转折/高潮定义） | 定义模糊，无法检验 |
| 张力曲线标注 | 无法检验，Agent 可以自己理解 |
| 从标注数据训练生成模型 | Agent 不需要训练，只需要参考 |
| 模板化自动生成 | 用户需要的是"糅合参考"，不是"模板套用" |
| 物品/怪物/能力独立检索 | 已由事件检索覆盖，单独设立性价比低 |
| Web UI | 所有操作通过 Agent CLI 和脚本完成 |

## 十一、成功标准

本项目是否成功，取决于用户能否在实际写作中：

1. **快速找到参考**：输入需求描述后，2 秒内返回匹配结果
2. **找到高质量参考**：返回的结果确实是用户需要的类型
3. **方便地糅合**：返回的结构化数据让 Agent 能轻松分析糅合
4. **覆盖足够广**：1000 本小说提供充足的参考池
5. **持续扩展**：新小说入库后，自动进入可检索状态

---

## 十二、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| LLM 章级分析质量不稳定 | 检索精度下降 | 引入质量校验脚本，低质量章节标记重做 |
| 向量搜索语义不准 | 匹配度低 | 选择合适的 embedding 模型，针对中文优化 |
| 数据规模增长超预期 | 性能下降 | 定期评估 pgvector → Qdrant 迁移必要性 |
| 检索结果同质化 | 参考价值低 | 引入多样性排序（不同类型/作者分散） |
