# 事件拆分方案反思

## 日期
2026-04-27

---

## 一、问题发现

### 用户反馈
"所有的问题根源只有一个：事件的边界定义不清晰。"

### 问题表现

在运行 pipeline 过程中，事件拆分产出存在以下问题：

| 问题层面 | 具体表现 |
|----------|----------|
| 字段边界 | Schema 定义与实际产出不一致（chapters vs sections，lines vs line_ranges） |
| 粒度边界 | 同为"主线事件"，粒度差距可达33倍（400行 vs 12行） |
| 覆盖边界 | 同一段文本被多个事件覆盖，但没有层级关系定义 |
| 校验结果 | h1x2 的30个事件文件全部校验失败 |

---

## 二、问题根源分析

### segmentation-rules.md 的定义

```
事件单元 = 一个完整戏剧动作
可回答: 谁/目标/过程/结果
切分点: 角色状态变化 | 目标变化 | 关系变化 | 结果出现 | 线索切换
```

### 定义缺失

| 缺失内容 | 导致的问题 |
|----------|-----------|
| 粒度锚点 | 没有定义"事件的最小/最大单元" |
| 层级关系 | 没有定义"主线事件与子事件的关系" |
| 重叠处理 | 没有定义"同文本被多线索覆盖时如何标记" |
| 结构角色 | 没有定义"转折/高潮等标签的判断标准" |

### 核心矛盾

**"戏剧动作"是一个文学概念，不是可程序检验的客观标准。**

LLM 对这个概念的理解必然有偏差，且偏差无法通过校验脚本发现。

---

## 三、最高目标确认

### 用户目标

"通过场景拆分，完整拆出一本小说的发展脉络，然后通过对很多小说的分析，最终实现使用 Agent 生成新小说的大纲、细纲、章纲，最终实现自动化小说写作工具。"

### 目标分析

| 链条环节 | 问题 |
|----------|------|
| 小说 → 拆事件 | 边界定义不清晰 |
| 标注结构角色 | 结构范式（转折/高潮）定义本身模糊 |
| 归纳规律 | 需要大量高质量标注，但标注质量不可控 |
| Agent生成 | 规律是否真的能指导生成？还是Agent主要靠自身能力？ |

### 关键发现

**Agent生成小说大纲，可能不需要从标注数据"学习规律"。**

Agent 本身有能力：
- 从阅读过的海量文本中隐式学习模式
- 根据用户指令生成结构
- 参考具体样例调整输出

**标注数据的定位应该是"检索参考"，不是"学习材料"。**

## 四、方案评估

### 方案A：坚持事件拆分

| 评估 | 结论 |
|------|------|
| 可控性 | 不可控：边界定义模糊，LLM理解偏差无法检验 |
| 投入产出 | 投入大，产出有限 |
| 结论 | 不推荐 |

### 方案B：改用场景拆分

| 评估 | 结论 |
|------|------|
| 边界定义 | 比"事件"更客观（地点切换、时间跳跃、焦点切换） |
| 数量膨胀 | 实测：网络小说每章平均1-3个场景，不膨胀 |
| 用户反馈 | "很难从中提取规律" |
| 结论 | 场景切分本身不产生规律，只是把文本切得更细 |

### 方案C：只做章节切分+摘要

| 评估 | 结论 |
|------|------|
| 可控性 | 完全可控：章节切分是机械操作 |
| 摘要质量 | 半可控：LLM偏差，但摘要不要求精确 |
| Agent可用性 | 能看到开局范式、推进范式、转折范式、节奏范式 |
| 结论 | 最简可行方案 |

## 五、核心结论

### 拆分悖论

**拆分越精细，边界越难定义；拆分越粗，规律提取越依赖理解。**

| 拆分层级 | 边界困难 | 规律提取困难 |
|----------|----------|--------------|
| 事件 | 戏剧动作模糊（不可控） | 需要理解事件角色 |
| 场景 | 焦点切换信号（半可控） | 需要理解场景关系 |
| 章节 | 无困难（完全可控） | 需要理解章节内容 |

### 规律提取的本质

规律不是从拆分中自动产生的，而是从**理解内容**中产生的。

例如：
- "主线中插入配角视角，制造悬念"——这是叙事技法，需要理解场景之间的关系
- "开局困境→主动自救"——这是开局范式，需要理解人物状态变化
- "家庭团聚vs家破人亡"——这是对比手法，需要理解情节含义

这些理解，Agent 可以自己完成，不需要人为标注。

## 六、建议方案

### 最简方案

**只做章节切分 + 内容摘要。**

| 操作 | 可控性 | 产物 |
|------|--------|------|
| 章节切分 | 完全可控（机械） | 章节文本 |
| 内容摘要 | 半可控（LLM偏差） | 每章50-100字摘要 |
| 人物名提取 | 可控（名词检测） | 出场人物列表 |

### 不做的内容

- 不做事件拆分（边界不可控）
- 不做场景拆分（用户反馈难提取规律）
- 不做结构角色标注（转折/高潮定义模糊）
- 不做张力曲线标注（无法检验）

### Agent如何使用产物

Agent从章节序列+摘要中可以看到：

| 能看到 | 用途 |
|--------|------|
| 第一章摘要：入狱→困境→自救思路 | 开局范式 |
| 第1-5章摘要：破案过程 | 推进范式 |
| 第7章摘要：家人团聚+配角自杀悬念 | 转折范式 |
| 情节跨越的章节数 | 节奏范式 |

**Agent 自己有能力从摘要序列中理解结构，不需要人为标注。**

## 八、讨论记录

### 关键对话节点

1. 用户指出问题根源："事件的边界定义不清晰"
2. 分析后发现问题是复合的：字段、切分、覆盖、定义多层问题
3. 用户澄清最高目标：自动化小说写作
4. 分析发现 Agent 可能不需要从标注数据学习规律
5. 评估场景方案后，用户反馈"很难从中提取规律"
6. 最终倾向最简方案：章节切分+摘要

### 核心共识

- 事件边界定义是不可控的
- Agent 主要靠自身能力，标注数据是参考不是学习材料
- 拆分越精细越难，越粗越依赖理解
- 最简方案可能是最可行的
