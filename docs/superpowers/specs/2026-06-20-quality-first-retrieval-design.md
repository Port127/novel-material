# 质量优先的小说素材检索系统设计

**状态：** 待用户书面审阅

**日期：** 2026-06-20

**目标版本：** Novel Material V2 检索层增强

## 1. 决策摘要

本项目继续定位为“小说写作参考检索库”，只负责检索与结构化展示，不在仓库内增加自动写作或问答生成。外部 Agent 负责理解、比较、糅合和生成。

检索设计采用“质量优先”原则：

1. 保留现有 4096 维向量，不把降维作为前置条件。
2. 当前规模先使用 4096 维精确向量检索建立质量基线。
3. 通过中文词法召回、结构化过滤、向量召回、结果融合、上下文补全和深度重排提升最终结果质量。
4. 用户可接受单次检索最多约 3 分钟，因此不以 2 秒作为硬性约束；但每个阶段仍需记录耗时并设置超时，避免无边界等待。
5. 当规模增长到 500～5000 本小说时，只有在性能基准确认精确搜索不可接受后，才启用近似候选索引；完整 4096 维向量仍用于候选集精排。
6. 任何加速方案必须通过质量门槛，不能以速度换取不可见的召回损失。

## 2. 当前项目核查结论

### 2.1 数据与运行状态

- 本地共有 11 本小说，`meta.yaml` 合计约 13,229 章。
- `chapters.yaml` 中现有约 13,100 条章级分析记录。
- 现有约 11,706 条章节向量，维度均为 4096；其中一本小说没有章节向量，一本失败素材缺少约 129 章分析结果。
- `data/novels/` 当前约 400 MB。
- 测试基线为 `1 failed, 49 passed, 1 skipped`，失败项是 `word_count` 契约不一致。
- 工作区已有用户修改：`config/providers.yaml` 将模型从 `qwen3.6-plus` 改为 `qwen3.7-plus`。后续实现不得覆盖该修改。

### 2.2 检索实现问题

1. `search_*` 函数负责查询和打印，却不返回结果；Typer CLI 把它们当作返回列表的服务函数，导致底层打印结果后 CLI 又误报“未找到匹配结果”。
2. 正式 `nm search` 只暴露 `outline / character / chapter / world`，文档中的 `event` 和内部已有的 `detail` 未注册。
3. `nm search chapter` 没有暴露 `--semantic`，实际默认走关键词分支。
4. 关键词检索使用 `%query%` 的 `ILIKE`，没有中文分词、倒排索引或相关度排名。
5. 关键词与向量是二选一分支，不是混合召回。
6. 所有向量列都是 `vector(4096)`；向量索引被注释，语义查询走精确全表扫描。
7. embedding 配置声明了维度，但 API 调用没有统一传递或校验实际维度，也没有记录模型、维度、文本构造方式和版本。
8. 当前没有 `tests/search/`、检索基准集、召回/排序指标、性能基准和失败案例回归。
9. 检索结果没有统一结构化协议，Agent 只能解析面向人的终端文本。

## 3. 目标与非目标

### 3.1 目标

- 让 Agent 通过稳定的结构化接口检索章节、事件、人物、世界观、大纲和细纲。
- 章纲和事件 Top 10 人工相关率不低于 80%。
- 返回结果兼顾语义相关、专名精确、结构条件和跨小说多样性。
- 返回前后章摘要与原文定位信息，帮助 Agent 理解上下文。
- 建立可重复的检索质量评测，使 embedding、提示词、融合权重和加速方案都能客观比较。
- 支持从当前 1.3 万章演进到 500 本、1000 本和 5000 本小说。
- 在 250 万章代表性数据集上，质量模式 P95 总耗时不超过 180 秒。

### 3.2 非目标

- 不增加自动写作、模板生成或最终回答生成。
- 不拆分场景或戏剧动作，章节仍是最小分析单元。
- 不训练生成模型。
- 不建设 Web UI。
- 不在没有评测证据时更换向量数据库或降低 embedding 维度。
- 不为追求固定的 2 秒响应牺牲质量。

## 4. 方案比较

### 方案 A：始终使用 4096 维全库精确搜索

优点是向量召回没有近似索引损失，架构简单。缺点是数据达到约 250 万章时，每次查询都要比较全部高维向量，多 Agent 并发时延迟和资源占用不可控。适合作为质量基线，不适合作为唯一的长期执行路径。

### 方案 B：立即降维并使用 HNSW

优点是索引成熟、速度快、存储较小。缺点是必须重建现有向量，且降维与近似索引都可能改变召回结果。用户明确质量优先，因此不采用为默认方案，只保留为未来有评测数据后的实验选项。

### 方案 C：分层质量检索（推荐）

保留 4096 维完整向量。当前规模使用精确向量召回；同时加入中文词法召回和结构化召回。各路候选先融合，再使用完整字段、邻章上下文和高质量重排器精排。未来规模过大时，只把近似索引用作“宽候选生成器”，候选仍由完整 4096 维和深度模型重新评分。

该方案的核心不是“永远不用索引”，而是“索引只能缩小候选，不能取代质量评测和最终精排”。

## 5. 推荐架构

```text
用户/Agent 查询
    ↓
SearchRequest（查询、类型、过滤条件、模式、数量、时间预算）
    ↓
查询规划
    ├── 词法召回：中文分词 + 专名/短语匹配
    ├── 结构召回：题材、标签、张力、章节位置、人物、场景等
    └── 语义召回：4096 维向量；当前精确搜索，未来可加候选索引
    ↓
候选标准化与融合（RRF，保留各路分数和命中原因）
    ↓
去重与多样性控制（限制单本小说占比）
    ↓
上下文补全（前后章摘要、原文行号、人物/世界观关联）
    ↓
深度重排（可插拔：规则 / Cross-Encoder / LLM）
    ↓
SearchResponse（Top N 结果 + 来源 + 分数 + 降级信息 + 耗时）
```

### 5.1 统一检索协议

新增与 CLI 展示解耦的服务协议：

- `SearchRequest`：`query`、`document_types`、结构化过滤、`limit`、`candidate_limit`、`mode`、`time_budget_seconds`。
- `SearchResult`：统一标识、素材 ID、文档类型、标题、摘要、元数据、原始章节位置、邻章信息、各路分数、最终分数和命中原因。
- `SearchResponse`：结果列表、实际执行阶段、各阶段候选数、耗时、使用的 embedding 版本和降级原因。

搜索模块只返回数据；CLI 负责 Rich 表格或 JSON 序列化。`--json` 是 Agent 的稳定入口，终端表格继续服务人工使用。

### 5.2 三路候选召回

**词法召回**用于小说名、人物名、宗门名、招式名和精确事件短语。中文文本先使用项目已有 `jieba` 分词，再构建 PostgreSQL `tsvector`/GIN 索引；对名称和短字符串保留 `pg_trgm` 或精确匹配兜底。

**结构召回**直接利用已有的题材、章节功能、张力、情绪、场景、技法、钩子、人物、结构角色等字段。过滤条件不是附属能力，而是小说素材检索的重要质量来源。

**语义召回**继续使用 Qwen 4096 维 embedding。章节检索文本不能只依赖 `summary`，需要评测以下构造方式：

1. `summary`（现有基线）；
2. `summary + key_event + plot_progress`；
3. 上述内容再加章节功能、情绪、场景、技法和钩子标签。

最终构造方式由检索评测决定，不能凭直觉直接替换现有向量。

### 5.3 融合、重排与多样性

- 首选 RRF 融合不同召回排名，避免直接比较不可通约的余弦分数、词法分数和规则分数。
- 默认候选池建议从 100～300 条起步，由评测确定；深度模式允许扩大候选池。
- 第一阶段重排读取完整结构字段；第二阶段可让 Cross-Encoder 或 LLM 同时阅读查询与候选。
- 深度重排失败时退回融合排序，不能返回空结果。
- 当至少有足够多不同素材可用时，Top 10 至少覆盖 5 本小说，单本小说默认不超过 3 条；比较指定小说时可显式关闭该限制。

### 5.4 上下文与原文定位

章节结果补充前一章和后一章摘要，跨章事件允许扩展相邻章节，但不创建“事件片段”或“场景片段”作为新分析单元。结果携带 `chapter_index.yaml` 中的章节号、标题、`start_line` 和 `end_line`，Agent 可按需读取 `source.txt` 精确定位。

### 5.5 规模演进策略

- **当前约 1.3 万章：** 使用 4096 维精确搜索，建立可信质量基线。
- **约 25 万章（500 本）：** 运行精确搜索的 P50/P95、内存和并发基准；未超过 180 秒预算则不引入 ANN。
- **约 50 万章（1000 本）：** 对二值量化、子向量索引或独立候选引擎做离线实验，但完整向量仍保留。
- **约 250 万章（5000 本）：** 若精确召回不可接受，使用高召回候选索引取较宽候选集，再以完整 4096 维精排。候选索引只有在 `Recall@候选集 ≥ 98%` 且最终 `nDCG@10` 相对精确基线下降不超过 1% 时才能启用。

检索后端通过接口隔离；只有 PostgreSQL 方案在代表性数据上无法同时满足 180 秒和质量门槛时，才评估 Qdrant、Milvus 或其他后端。

## 6. 数据与数据库演进

第一阶段不迁移向量、不降低维度，只修复服务契约和建立评测。

第二阶段为现有实体表增加可重建的词法索引字段及索引，数据由现有 YAML/数据库字段本地回填，不调用 LLM。embedding 缓存增加模型、维度、输入文本构造版本和生成时间等 provenance；同步时校验实际维度与数据库字段一致。

未来如需候选索引，优先在保留原 `vector(4096)` 的情况下增加表达式索引或旁路候选索引，不覆盖现有向量。所有检索派生数据都必须能从 YAML 和 embedding 配置重建。

## 7. 失败与降级策略

- embedding 服务不可用：返回词法 + 结构化结果，并在 `SearchResponse` 标记语义召回降级。
- 深度重排超时或失败：返回融合排序结果。
- 部分素材缺向量：仍参与词法和结构召回，不静默消失。
- 数据库不可用：CLI 非零退出并给出明确诊断，不打印“未找到”。
- 某一路召回为空：其他召回继续执行，并记录候选数。
- 时间预算将耗尽：停止后续重排，返回当前最佳结果和未完成阶段。
- 任何异常都不能被转换为误导性的“没有匹配素材”。

## 8. 评测与验收

### 8.1 Golden Query 数据集

从需求文档中的真实查询扩展出人工标注集，覆盖精确型、探索型、比较型、反套路型、跨题材型和多条件型查询。每条查询记录：

- 查询文本与检索类型；
- 结构化过滤条件；
- 相关结果及 0～3 级相关性；
- 明确不相关的困难负例；
- 是否需要多样性和邻章上下文。

优先完成章纲和事件检索，再扩展人物、大纲、世界观和细纲。失败查询进入固定回归集。

### 8.2 指标

- 候选召回：Recall@50、Recall@100、MRR。
- 最终排序：nDCG@10、Precision@10。
- 业务标准：章纲/事件相关率 ≥ 80%，大纲/人物 ≥ 70%，世界观 ≥ 60%。
- 多样性：在候选充足时，Top 10 不少于 5 个不同 `material_id`。
- 完整性：有章节结果时，原文定位字段覆盖率 100%；非边界章节的邻章摘要覆盖率 100%。
- 稳定性：空结果率、降级率、异常率。
- 性能：各阶段 P50/P95、总 P95、候选规模、数据库缓冲命中和并发吞吐。

### 8.3 质量门禁

任何 embedding 输入变更、模型变更、融合策略、候选索引或数据库迁移都必须：

1. 与当前 4096 维精确搜索基线对比；
2. 通过固定 Golden Query；
3. 输出逐查询差异，不能只看平均分；
4. 最终 nDCG@10 不得无解释下降；
5. 加速方案必须额外报告候选召回损失。

## 9. 实施边界与阶段

本设计后续拆成三个顺序执行、可独立验收的实施阶段：

1. **检索可靠性与评测基线：** 修复返回契约、补齐 CLI/JSON、建立搜索测试和 Golden Query，记录当前 4096 维基线。
2. **质量优先混合检索：** 增加中文词法索引、结构召回、RRF、多样性、邻章与原文定位，保持精确 4096 维召回。
3. **深度重排与规模验证：** 接入可插拔重排器，建立 25 万/50 万/250 万章性能基准，只有通过质量门禁才启用候选索引。

每个阶段都必须先写失败测试、实现最小闭环、运行质量回归和项目测试，再进入下一阶段。

## 10. 外部资料与采用原则

- [pgvector 官方文档](https://github.com/pgvector/pgvector)：精确/近似检索、HNSW 参数、过滤、二值量化、子向量索引、混合检索和“候选后用完整向量重排”的做法。
- [pgvector-python 混合检索示例](https://github.com/pgvector/pgvector-python/tree/master/examples/hybrid_search)：并行词法/语义召回以及 Cross-Encoder 重排。
- [PostgreSQL 全文索引](https://www.postgresql.org/docs/current/textsearch-indexes.html)：GIN/GiST 全文检索索引的官方说明。
- [PostgreSQL pg_trgm](https://www.postgresql.org/docs/current/pgtrgm.html)：短语、名称和模糊字符串匹配的索引能力。
- [jieba](https://github.com/fxsjy/jieba)：项目已有的中文分词依赖。
- [Sentence Transformers Retrieve & Re-Rank](https://www.sbert.net/examples/sentence_transformer/applications/retrieve_rerank/README.html)：先宽召回再用 Cross-Encoder 精排的社区实践。
- [BEIR](https://arxiv.org/abs/2104.08663)：异构检索任务评测与“不能只依赖单一检索方式”的研究背景。

外部资料只提供候选技术路线，不直接替代本项目评测。所有参数、模型和后端选择最终以小说素材 Golden Query 和实际规模基准为准。
