# LLM 响应契约统一加固设计

## 背景

世界观提取在现实题材中收到 `power_system: []` 后，把数组当作对象调用 `.get()`，导致完整流水线中断。对全部 `call_llm()` 业务消费点的定向审查进一步确认：单章分析、大纲前提、标签生成和素材分类也存在同类高风险边界；总体评估、大纲分幕、beats 和人物提取虽然通常会被上层兜底，但缺少明确的 schema 诊断。

根因不是 JSON 解析失败，而是 `call_llm()` 只保证返回合法 JSON，业务层却把合法 JSON 等同于符合业务契约的对象。

## 目标

- 每个 LLM 业务入口在读取、遍历、落盘或写数据库前，显式验证自己的响应契约。
- 对可安全兼容的空值或既有裸数组格式做有限归一化，不静默吞掉非空错误内容。
- 结构错误进入现有阶段级、批次级或单项级容错，不以未处理的 `AttributeError`、`TypeError` 中断流水线。
- 日志区分 API 调用错误、JSON 解析错误和业务 schema 错误。
- 不新增 LLM repair 调用，不产生额外 API 消耗。

## 非目标

- 不把所有业务 schema 硬编码进通用 `call_llm()`。
- 不切换模型、服务商或 `response_format`。
- 不调整 embedding、数据库 schema、检索算法或产物字段含义。
- 不借机重构无关流水线代码。

## 方案比较

### 方案一：各调用点增加零散 `isinstance`

改动最小，但规则容易重复和漂移，错误信息不统一，也难以独立测试。

### 方案二：为全部响应建立严格 Pydantic 模型

类型边界最强，但现有响应兼容形状较多，一次性建模会显著扩大改动，并可能把可恢复的历史格式全部变成硬失败。

### 方案三：业务专属纯契约函数加少量共享原语

在通用层提供 `LLMResponseContractError` 及基础对象、数组检查；每个业务模块定义自己的归一化和字段规则。这样既统一错误语义，又保留各阶段不同的兼容与降级策略。本次采用此方案。

## 架构

### 通用契约原语

新增轻量模块，职责仅包括：

- `LLMResponseContractError`：统一表示合法 JSON 不符合业务 schema。
- 要求顶层或字段为对象、数组、字符串、数值的基础检查。
- 过滤对象数组时不静默接受错误元素；调用方必须选择“整体拒绝”或“记录并跳过”。

通用原语不认识 `power_system`、`chapters`、`quality` 等业务字段，也不负责日志、重试或兜底。

### 业务契约函数

每个入口在所属模块中保留纯函数，完成该响应的字段级验证与有限归一化：

```text
call_llm
  → JSON 解析
  → 业务契约函数
  → 领域容错策略
  → 遍历 / 落盘 / 数据库写入
```

业务契约函数不得执行文件或数据库写入，以便单元测试精确覆盖。

## 字段兼容原则

- 缺失字段、`null` 和契约明确允许的空集合，可以归一化为标准空值。
- 历史上明确支持的裸数组响应继续兼容，例如批量章节、幕序列、beats 和人物列表。
- 对象字段出现空数组时，只有该字段语义明确表示“维度不存在”才允许归一化为 `{}`。
- 对象字段出现非空数组、字符串或数值时必须报 schema 错误，禁止挑第一项或丢弃内容。
- 数组元素必须是对象或指定标量类型；错误元素不得在无诊断情况下进入产物。
- 未知顶层字段可以保留或忽略，但不能影响已知字段校验。

## 分阶段设计

### 第一组：会逃出局部容错的高风险入口

#### 世界观

- 提示词改为逐字段声明标准空结构：`power_system/geography/lore` 为对象，`factions` 为数组。
- `power_system/geography/lore` 的缺失、`null` 或空数组归一化为 `{}`；非空错误类型拒绝。
- `factions` 的缺失或 `null` 归一化为 `[]`，其他非数组类型拒绝。
- schema 错误复用现有空结构兜底，写入 `llm_success: false`，不新增重试。

#### 单章分析

- 顶层必须是对象。
- 对摘要、张力、人物和章节级列表字段执行类型校验；不把字符串等错误集合静默转成数组。
- 契约验证、质量验证、规范化和保存纳入同一个单章错误隔离边界。
- 无效结果不写章节 YAML，记录当前章失败，交给既有缺失章节重分析流程补齐。

#### 大纲前提

- 顶层必须是对象，`premise/structure_type` 为字符串，`theme/tone` 为字符串数组，`total_acts` 为正整数。
- schema 错误与 API 错误一样使用现有默认前提，确保 `save_meta_with_premise()` 只接收标准对象。

#### 标签生成

- 在建立数据库连接前验证完整响应。
- `genre_primary/genre_secondary/elements/style/hooks/tropes/themes` 只接受契约规定的字符串或字符串数组；`setting/structure` 为可空字符串；`genre_description` 为字符串。
- schema 错误使用现有默认标签兜底，禁止边验证边写候选表。

#### 素材分类

- 保留顶层对象检查，补齐 `quality` 对象及 `writing/plot/character` 数值校验。
- 所有结构错误统一转换为 `LLMResponseContractError`，由调用方返回现有 `status: failed` 结果。

### 第二组：已有上层兜底但诊断不明确的入口

#### 总体评估

- 验证顶层对象以及 `novel_type/core_characters_hint` 字符串数组。
- schema 错误保留现有阶段失败行为，但日志明确标记为响应契约错误。

#### 大纲分幕与 beats

- 继续兼容裸数组和对象包裹数组。
- 校验数组元素、章节范围、编号和必要文本字段。
- 分幕错误继续使用简单划分；beats 错误继续按序列跳过并计入失败数。

#### 人物提取

- 继续兼容裸数组和 `characters` 包裹格式。
- 校验每个人物为对象且名称属于候选名单；嵌套关系和关键事件必须是对象数组。
- schema 错误继续使用出场统计基础档案兜底。

### 已有可靠防护的入口

- insights 保留 `_coerce_items()`、字段验证和 repair 后复验。
- 搜索重排保留 `_validate_rankings()`。
- 审计复核保留 `ReviewDecision.model_validate()`。

本次只补充必要的回归测试，不重写这些成熟入口。

## 错误与日志语义

统一使用以下分类：

- API/网络错误：现有连接、超时和服务端错误分类。
- JSON 解析错误：现有 `JSONDecodeError` 重试路径。
- `schema_invalid`：JSON 合法，但不符合业务响应契约。

错误日志至少包含业务上下文、字段路径、期望类型和实际类型；不得记录完整 prompt、原文或未经脱敏的完整响应。

## 测试策略

### 通用原语单元测试

- 顶层对象和数组校验。
- 字段路径错误信息。
- `bool` 不得被当作普通数值接受。

### 业务契约参数化测试

每个入口至少覆盖：

- 标准有效响应保持语义不变。
- 缺失、`null` 和允许的空集合被归一化。
- 顶层数组或标量的处理符合该入口兼容策略。
- 嵌套字段错误类型产生 `schema_invalid`。
- 数组中的非对象元素不会进入产物。

### 容错集成测试

- 世界观空数组不再复现 `.get()` 异常，并生成失败索引。
- 单章畸形结果不写文件、不终止其他章节处理。
- 大纲前提畸形结果使用默认值继续。
- 标签畸形结果在数据库写入前被拒绝。
- 素材分类畸形 `quality` 返回结构化失败。
- 第二组入口保持原有简单大纲、跳过序列或基础人物兜底行为。

## 交付顺序

1. 建立通用契约异常、原语及测试。
2. 逐个修复第一组五个高风险入口，每个入口独立完成红—绿测试循环。
3. 修复第二组入口的契约和诊断语义。
4. 运行定向测试、完整测试和静态检查。
5. 使用现有素材状态执行 `nm pipeline continue nm_novel_20260623_70br --mode standard`；该运行会产生 API 消耗，必须在代码测试通过后单独征得用户确认。

## 成功标准

- 所有 `call_llm()` 业务消费点在首次 `.get()`、遍历、落盘或数据库写入前具有显式响应契约。
- 审查报告中的 5 个 Critical 与 2 个 Suggestion 均有自动化回归测试。
- 畸形合法 JSON 不再以未处理的 `AttributeError`、`TypeError` 逃出既定容错边界。
- 正常响应产物格式保持兼容，不引入额外 LLM 调用。
