# 分层分析与质量报告设计

## 1. 背景

当前完整流水线已经能够生成章节、大纲、世界观、人物和标签等产物，但真实运行暴露出四类相互关联的问题：

1. 流水线结束后只有耗时、Token、API 调用等基础计数，没有解释运行是否健康、哪些阶段发生降级、产物是否可用。
2. 世界观由一次 LLM 调用从章级摘要池中抽取固定四类结构，缺少题材适配、实体关系、章节证据和质量门禁。
3. 人物字段名义上包含弧线、心理和关系，但核心人物可能退化为统计兜底档案。最近一次 1084 章素材中，主角档案只剩“出场 1026 章，为主要角色之一”，弧线、心理和关系均为空。
4. 总体评估执行五批采样调用，却主要只在 `--window` 模式中被压缩为一段全局上下文，投入与复用价值不匹配。

这些问题不能靠单独加长提示词解决。需要把全局理解、专题抽取、后置综合、质量审计和运行报告组织成一条分层分析链路。

## 2. 目标与非目标

### 2.1 目标

- 将总体评估拆为前置全局导航和后置作品画像，使全局分析结果被章节、人物、世界观及写作 Agent 复用。
- 为 5–12 名主要人物生成可用于构思人物的完整小传，其余人物保留可检索简档。
- 将世界观升级为题材自适应的分层产物，包含概览、实体、关系和章节证据。
- 在流水线结束时同时输出运行健康报告和产物质量报告。
- 默认先做确定性检查，只对可疑项进行定向 LLM 复审；最终审计保持只读。
- `standard` 模式相对现有可比流程的额外耗时不超过 10%。达到预算后停止 LLM 复审，但继续输出完整的规则审计结果。
- 保持旧素材、现有命令和 PostgreSQL 查询层可渐进迁移。

### 2.2 非目标

- 不在本轮建设完整小说知识图谱。
- 不让最终报告成为新的事实来源；报告只汇总运行事件和已有分析产物。
- 不在最终审计阶段静默修改 YAML 或产生无上限 API 消耗。
- 不以“字段更长”替代证据、覆盖率和跨文件一致性检查。
- 不宣称新的分析产物提升了检索质量，除非后续人工 Golden Query 与回归评测能够证明。

## 3. 方案选择

### 3.1 未采用：局部增强

分别扩展世界观提示词、人物字段和 `evaluation.yaml`，最后拼接报告。该方案改动较小，但各阶段仍会重复理解小说，证据与质量语义继续割裂，无法可靠识别“字段存在但实际为兜底”的情况。

### 3.2 采用：分层分析流水线

在现有 YAML 事实层和阶段编排上增加前置导航、后置作品画像、产物审计与最终报告。专题分析继续保持独立模块，通过稳定产物契约传递上下文。

### 3.3 暂缓：小说知识图谱

统一建模人物、地点、势力、事件、规则及其关系，长期推理能力更强，但需要重做大量存储、同步和搜索契约，超出当前问题的必要范围。

## 4. 总体架构

```text
ingest
  ↓
evaluate：前置全局导航
  ↓
analyze → insights
  ↓
outline / worldbuilding / characters / tags
  ↓
refine
  ↓
profile：后置作品画像
  ↓
audit：产物质量审计
  ↓
validate → sync
  ↓
report：运行情况与产物质量汇总
```

### 4.1 阶段职责

| 阶段 | 职责 | 主要输入 | 主要输出 |
|---|---|---|---|
| `evaluate` | 生成后续分析使用的全局导航 | 分层采样原文、章节索引 | `evaluation.yaml` |
| `analyze` | 生成稳定章级事实与摘要 | 原文、可选全局导航、可选前后章上下文 | `chapters.yaml` |
| `worldbuilding` | 生成题材适配的世界观实体和关系 | 章级分析、全局导航 | `worldbuilding/` |
| `characters` | 筛选人物层级并生成完整小传或简档 | 章级分析、全局导航 | `characters/` |
| `profile` | 形成面向写作 Agent 的作品级入口 | 全部稳定分析产物 | `work_profile.yaml` |
| `audit` | 检查产物完整性、覆盖率、证据和一致性 | 全部稳定分析产物、运行预算 | 内存审计结果、报告输入 |
| `report` | 汇总运行健康与产物质量 | 运行事件、审计结果 | Markdown、YAML、终端摘要 |

### 4.2 `evaluate` 与 `--window` 解耦

- `evaluate` 负责生成全局导航，不再由 `--window` 隐式触发。
- `--window` 只控制章级分析是否使用前章摘要、当前章和下章预览。
- `analyze` 在存在 `evaluation.yaml` 时消费全局导航；缺失时明确降级，但仍可运行。
- `standard` 和 `deep` 的完整流水线默认执行 `evaluate`；`fast` 默认跳过。
- 独立执行 `nm pipeline evaluate <id>` 的能力保留。

## 5. 前置全局导航

`evaluation.yaml` 升级到新 schema，包含：

```yaml
schema_version: 3.0.0
novel_type: []
premise: ""
main_thread_summary: ""
stage_map:
  - stage: opening
    chapter_ranges: []
    central_conflict: ""
    turning_points: []
core_character_candidates:
  - name: ""
    reasons: []
    confidence: 0.0
worldbuilding_dimensions: []
analysis_focus: []
sample_coverage:
  sampled_chapters: []
  covered_ranges: []
  limitations: []
```

设计约束：

- `stage_map` 表达采样推断，不冒充完整章级事实。
- 核心人物候选必须包含入选原因和置信度，供人物筛选器使用。
- 世界观维度只声明后续应分析什么，不在前置阶段生成完整世界观。
- `sample_coverage.limitations` 必须说明采样无法覆盖的部分。
- 所有字段阈值进入 `src/novel_material/schema/fields.yaml` 或对应集中配置，不在业务代码和多份文档中重复硬编码。

## 6. 人物分层小传

### 6.1 人物选择

主要人物不再只按出场章数判断。选择器综合以下信号：

- 前置全局导航中的核心人物候选。
- 章级关键事件参与度。
- 出场频率和贯穿范围。
- 人物关系中心度。
- 主角、对立角色、关键推动者等叙事功能。

完整小传目标数量为 5–12 人。该范围作为集中配置和审计契约管理。其他人物生成简档，不因数量过多无限扩大 LLM 请求。

### 6.2 完整小传

主要人物档案至少包含：

- 身份、角色层级、人物原型与道德倾向。
- 生平概要。
- 外在欲望、内在需求、恐惧、缺陷和性格矛盾。
- 分阶段人物弧线。
- 关键关系及关系变化。
- 行为习惯、语言特征和典型互动模式。
- 关键场景与章节证据。
- 可借鉴的塑造方法及适用边界。
- 每项分析的置信度，以及 `fact` 或 `inference` 标记。

### 6.3 简档与兜底

非主要人物简档包含身份、叙事功能、出场范围、主要关系和代表事件。统计兜底可以用于简档，但不得把主要人物的统计兜底标记为成功的完整小传。若主要人物缺少弧线、心理、关系或证据，审计至少产生 `error`。

## 7. 分层世界观

世界观目录调整为：

```text
worldbuilding/
├── _index.yaml
├── overview.yaml
├── dimensions.yaml
├── entities/
│   └── *.yaml
└── relations.yaml
```

### 7.1 题材维度路由

世界观维度由全局导航、题材 profile 和章级信号共同决定。例如：

- 都市重生：时代环境、商业规则、社会阶层、法律与组织网络。
- 仙侠玄幻：力量体系、资源、势力、地理、历史和超自然规则。
- 悬疑：制度环境、空间结构、秘密组织、信息规则和案件背景。

系统不得为了满足固定 schema，要求都市小说虚构修炼等级，也不得因为没有超自然力量就判定世界观短或失败。

### 7.2 实体与关系

实体卡片包含类型、名称、描述、属性、重要性、首次和关键出现章节、证据与置信度。关系文件记录从属、冲突、联盟、依赖、约束和演化关系，并引用双方实体 ID 与章节证据。

`overview.yaml` 解释世界如何运行以及哪些设定真正驱动剧情；它不重复堆叠实体描述。

## 8. 后置作品画像

`profile` 阶段读取章节、大纲、人物、世界观和标签产物，不重新读取全书原文。输出 `work_profile.yaml`，包含：

- 核心卖点与读者期待。
- 故事结构和节奏模式。
- 人物群像与关系动力。
- 世界观驱动剧情的机制。
- 反复出现的母题和写作技法。
- 可迁移的创作启示及适用边界。
- 已知覆盖限制和低置信度结论。

该文件是写作 Agent 的作品级入口，但不是章节、人物或世界观事实文件的替代品。需要具体证据时，Agent 必须继续读取或检索下层产物。

## 9. 质量审计

### 9.1 审计条目

所有问题使用统一结构：

```yaml
code: character_profile_fallback
severity: error
artifact: characters/profiles/陈汉升.yaml
message: 主要人物使用了统计兜底档案
evidence:
  missing_fields: [arc_summary, psychology, relationships]
next_actions:
  - command: nm pipeline characters nm_xxx --repair-character 陈汉升
```

严重程度：

| 等级 | 含义 | 对流水线的影响 |
|---|---|---|
| `blocker` | 契约损坏、关键产物缺失或无法安全同步 | 阻止同步，运行失败或降级 |
| `error` | 明显质量失败，例如主要人物为空壳 | 保留产物，运行标记为 `degraded` |
| `warning` | 证据或覆盖不足，但产物仍可使用 | 允许同步 |
| `info` | 统计与非阻断优化建议 | 不影响状态 |

### 9.2 混合审计

1. 先执行确定性规则，检查 schema、空值、长度、数量、证据引用、覆盖率和跨文件一致性。
2. 只有规则无法判断的可疑项才进入 LLM 复审。
3. 单阶段生成后允许一次定向修复。
4. 最终审计只读，不再修改事实 YAML。
5. 超出时间或调用预算的待审项标记为 `not_reviewed_due_to_budget`，不得静默忽略。

### 9.3 时间预算

- `fast`：只执行确定性审计和报告，目标额外耗时 1–2 分钟以内。
- `standard`：相对现有可比流程额外耗时不超过 10%。达到预算后停止新的 LLM 复审。
- `deep`：允许更高的显式预算，但仍必须有配置上限。

以最近一次 1084 章素材为参考，已记录阶段耗时约 6 小时 23 分，其中章级分析约 5 小时 25 分。人物阶段当前处理 134 人耗时约 42.8 分钟。自适应筛选预计能够抵消新增导航、作品画像和定向复审的大部分耗时。该估算是设计基线，不是完成后的性能结论；实施后必须用相同素材复测。

## 10. 报告设计

### 10.1 文件布局

```text
data/novels/{material_id}/reports/
├── latest.md
├── latest.yaml
└── runs/
    └── {run_id}.yaml
```

- `runs/{run_id}.yaml` 是单次运行的不可变机器报告。
- `latest.yaml` 是当前最新报告的副本或稳定指针产物。
- `latest.md` 与 `latest.yaml` 语义一致，面向人工审阅。
- 报告不是 YAML 事实层的一部分，不被专题分析反向消费。

### 10.2 报告内容

运行报告包含：

- 总状态、开始和结束时间、总耗时。
- 阶段耗时、占比和状态。
- API 调用、错误、重试、Token 和可获得的成本信息。
- 预算上限、实际使用量和预算停止原因。
- 运行级、阶段级诊断及恢复命令。

产物质量报告包含：

- 章节与 insight 覆盖情况。
- 主要人物完整小传通过数量、简档数量和失败原因。
- 世界观启用维度、实体、关系和证据覆盖。
- 作品画像及其他关键产物状态。
- 按严重程度归类的问题、未审项和精确下一步命令。

### 10.3 终端摘要

终端只显示总状态、耗时变化、关键产物通过情况、最高优先级风险、报告路径和下一步命令。完整问题明细不重新刷满终端。

报告生成失败不得覆盖真实流水线结果，但本次运行需要增加可观测性降级诊断。

## 11. CLI 与运行模式

新增或扩展以下公开入口：

```bash
nm pipeline profile <material_id>
nm pipeline characters <material_id> --repair-character <name>
nm validate artifacts <material_id> [--review]
nm pipeline report <material_id> [--run-id <run_id>]
```

- `--repair-character` 可重复传入，只重建指定人物及受影响索引。
- `nm validate artifacts` 默认只做规则审计；`--review` 允许在配置预算内定向 LLM 复审。
- `nm pipeline report` 只根据现有事件和审计结果重建报告，不调用 LLM，不修改事实产物。
- 完整流水线根据运行模式自动编排 `profile`、`audit` 和 `report`。

最终 CLI 名称在实施时必须通过现有 Typer 命令风格和冲突检查，但不得改变上述职责边界。

## 12. 失败处理与状态语义

- 新产物先写临时文件，完成 schema 校验后原子替换正式文件。
- LLM 输出无法解析或不符合契约时最多修复一次；仍失败则记录原始诊断，不伪装成功。
- `blocker` 阻止同步；`error` 使运行至少为 `degraded`；`warning` 和 `info` 不阻断。
- 最终审计和报告失败不得删除或重写已成功的事实产物。
- `continue` 根据阶段状态和产物契约恢复，不仅依赖文件是否存在。
- 对失败、降级和中断运行都尽力生成报告；报告中明确数据是否完整。

## 13. 兼容与迁移

### 13.1 总体评估

- 新写入使用 `schema_version: 3.0.0`。
- 读取端为旧 `2.0.1` 结构提供适配器。
- 旧素材不会在读取时自动改写。

### 13.2 人物与世界观

- 旧人物档案继续可读；审计识别主要人物的统计兜底和缺失字段。
- 旧世界观四文件继续通过兼容加载器进入搜索和同步。
- 新世界观实体优先复用通用实体存储；关系若需要新表，必须走显式数据库迁移。
- 在人工检索基线完成前，不因新字段擅自改变现有 embedding 维度或宣称检索提升。

### 13.3 编排与断点

- `--window` 参数保留，仅移除其对 `evaluate` 的隐式触发职责。
- `full`、`continue` 和 `status` 识别新增阶段。
- 已完成旧流水线的素材可以单独执行 `profile`、审计和报告，不强制重跑章级分析。

## 14. 测试策略

### 14.1 单元测试

- 全局导航采样覆盖和旧 schema 适配。
- 人物自适应筛选、数量预算和关系中心度。
- 题材维度路由及不存在维度的合法空值。
- 证据引用、跨文件一致性和兜底检测。
- 审计等级映射、LLM 复审选择和 10% 时间预算停止条件。
- 报告聚合、路径、排序和敏感信息清理。

### 14.2 契约测试

- 提示词输出与 Pydantic/schema 契约一致。
- 新旧人物、世界观和总体评估格式都能被读取。
- CLI 文本和 JSON/机器输出保持稳定状态语义。

### 14.3 集成测试

- 小型固定小说走完整流水线，覆盖成功、降级、阻断和断点续传。
- 最终审计不修改事实 YAML。
- `blocker` 阻止同步，`warning` 允许同步。
- 成功、失败和中断运行都能生成一致报告。

### 14.4 回归与真实 smoke test

- 固定都市、仙侠和悬疑样例，验证世界观维度不会一刀切。
- 固定主要人物样例，验证完整小传不会退化为统计模板。
- 默认测试不调用真实 LLM；真实模型测试显式运行并记录模型、耗时、Token 和产物差异。

## 15. 分期交付

### 第一期：审计与报告

- 建立审计条目、规则检查、运行/产物报告和终端摘要。
- 接入现有产物，不改变事实文件格式。
- 以真实兜底人物档案作为回归样例。

### 第二期：前置导航与人物小传

- 升级 `evaluation.yaml` 并解除 `--window` 绑定。
- 实现自适应人物筛选、完整小传、定向修复和兼容读取。
- 用人物阶段耗时与质量回归验证标准模式预算。

### 第三期：世界观与后置作品画像

- 实现题材维度路由、世界观实体和关系。
- 增加 `work_profile.yaml`。
- 完成存储、同步和搜索适配及跨题材回归。

每一期都必须能独立交付、验证和回退。后一期不得以未验证的前一期质量假设为前提。

## 16. 验收标准

- 完整流水线结束后同时生成终端摘要、Markdown 报告和机器可读 YAML 报告。
- 报告可区分运行健康与产物质量，并给出可执行的恢复或定向重跑命令。
- 主要人物统计兜底不会再被当作完整小传成功；现有真实兜底样例能触发 `error`。
- 完整小传只覆盖自适应选择的 5–12 名主要人物，其余人物保持简档。
- 世界观产物包含题材维度、概览、实体、关系和章节证据；不适用维度允许为空且不构成质量失败。
- 前置导航和 `--window` 可以独立启用，旧 `evaluation.yaml` 仍可读取。
- 最终审计不修改事实 YAML；阶段内自动修复最多一次。
