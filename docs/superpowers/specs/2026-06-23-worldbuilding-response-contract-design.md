# 世界观响应契约修复设计

## 背景

世界观提示词要求 `power_system`、`geography`、`lore` 返回对象，却同时要求不存在的维度“留空数组”。现实题材可能因此返回 `power_system: []`，而落盘逻辑直接调用 `.get()`，触发 `AttributeError` 并中断流水线。

## 方案比较

1. 仅在索引统计处增加类型判断：改动最小，但错误结构仍可能写入 YAML，不能保证下游契约。
2. 仅修改提示词：可以降低异常概率，但 LLM 输出仍不具备强类型保证。
3. 修改提示词并在业务边界归一化、校验响应：同时消除契约歧义并阻止错误结构进入存储层，作为本次采用方案。

## 设计

### 响应契约

世界观响应的标准空结构为：

```json
{
  "power_system": {},
  "geography": {},
  "factions": [],
  "lore": {}
}
```

提示词必须逐字段说明空值类型，不再使用“所有维度统一留空数组”的表述。

### 归一化与校验

在 `call_llm()` 返回后、计算索引及写入文件前设置单一业务边界：

- 顶层必须是对象，否则抛出明确的类型错误。
- `power_system`、`geography`、`lore` 缺失、为 `null` 或为空数组时，兼容归一化为 `{}`。
- 上述对象字段出现其他错误类型，或出现非空数组时，抛出明确的类型错误，避免静默丢失内容。
- `factions` 缺失或为 `null` 时归一化为 `[]`；其他非数组类型视为契约错误。
- 合法字段内容原样保留，未知顶层字段不影响当前产物。

归一化函数保持纯函数，不负责日志、LLM 重试或文件写入，以便独立测试。

### 错误处理

结构错误由世界观阶段现有容错路径转换为空结构并记录错误，确保完整流水线可以继续，同时由索引中的 `llm_success: false` 和后续产物审计暴露降级状态。本次不增加第二次 LLM 调用，避免额外 API 消耗。

## 测试

新增世界观响应契约单元测试，覆盖：

- 现实题材常见的 `power_system: []` 被归一化为 `{}`。
- 缺失值和 `null` 被归一化为标准空结构。
- 正常响应保持不变。
- 顶层数组、非空对象字段数组及错误的 `factions` 类型被拒绝。
- 归一化后的空结构可以安全计算索引，不再复现 `.get()` 异常。

## 范围

本次只修改世界观提示词、响应边界和对应测试，不调整世界观存储格式、检索逻辑、模型配置或其他流水线阶段。
