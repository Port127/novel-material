## 一、项目定位

**novel-material 是一个小说写作参考检索库。**

它不是用来"学习规律"的训练系统，而是用来**提供检索参考**的素材库。

本项目只负责检索与结构化展示，不在项目内部生成用户最终采用的小说内容。参考的理解、比较、糅合和生成由外部 Agent 与用户共同完成。

**类比**：建筑师不会从 100 栋建筑中"学习"如何设计，而是翻阅优秀案例图集，把多个案例的精华组合到自己的设计中。本项目就是那个"案例图集"。

## 二、用户的两个写作场景

### 场景 A：前期规划阶段

用户在动笔之前，需要构建小说的整体框架。

**需要辅助生成的内容：**
- 世界观设定
- 规则体系（力量体系、社会规则等）
- 大纲（全书结构）
- 细纲（幕/序列/节拍级结构）

**用户工作流：**
1. 用户与 Agent 讨论，明确自己想要什么样的世界观/大纲
2. Agent 从本项目中检索 N 条匹配度最高的参考
3. 用户 + Agent 将多条参考糅合，形成自己的设定

### 场景 B：章节写作阶段

用户在写具体章节时，需要参考同类章节的写法。

**需要辅助生成的内容：**
- 章纲（单章节的结构和推进方式）
- 人物描写（人物的出场、互动、弧光展现）
- 事件描写（特定类型事件的写法）

**用户工作流：**
1. 用户写到某个章节/人物/事件时，与 Agent 讨论需要什么参考
2. Agent 从本项目中检索 N 条匹配度最高的参考
3. 用户 + Agent 将多条参考糅合，写出自己的内容

## 三、需求优先级与依赖关系

### 3.1 需求优先级

| 优先级 | 需求 | 理由 | 验收标准 |
|-------|------|------|---------|
| **P0** | 章纲检索 | 最高频使用，章节写作核心需求 | 查询"开局困境"返回结果相关率 > 80% |
| **P0** | 事件检索 | 场景 B 核心需求，写具体情节时必用 | 查询"雨中告别"返回结果相关率 > 80% |
| **P1** | 大纲检索 | 场景 A 核心需求，规划阶段必用 | 查询"废柴逆袭大纲"返回结果相关率 > 70% |
| **P1** | 人物检索 | 人物塑造关键参考 | 查询"导师型人物"返回结果相关率 > 70% |
| **P2** | 世界观检索 | 前期规划需求，使用频率较低 | 查询"修仙力量体系"返回结果相关率 > 60% |
| **P2** | 细纲检索 | 序列与节拍参考 | 查询"感情线节拍"返回结果相关率 > 70% |

**优先级定义依据**：
- P0：高频使用 + 核心业务场景（写章节时必须）
- P1：核心业务场景（规划阶段必须），但使用频率中等
- P2：辅助场景或待实现功能

### 3.2 需求依赖关系

```
章级分析（analyze）
    │
    ├──→ 章纲检索（search chapter）
    │
    ├──→ 事件检索（search event）
    │
    └→ 人物出场数据 ──┐
                      │
大纲生成（outline） ──┼──→ 大纲检索（search outline）
                      │
世界观提取（worldbuilding）──→ 世界观检索（search world）
                      │
人物提取（characters）───────┴──→ 人物检索（search character）
```

**设计影响**：
- 章级分析是最核心的依赖，影响 3 个检索需求
- 人物检索同时依赖章级分析（出场数据）和人物提取（档案数据）
- 细纲检索依赖大纲生成的序列/节拍数据

### 3.3 需求验收标准

| 需求 | 验收标准 | 度量方式 |
|------|---------|---------|
| 世界观检索 | 查询"修仙力量体系"返回结果中至少 60% 相关 | 人工标记相关/不相关 |
| 大纲检索 | 查询"废柴逆袭大纲"返回结果中至少 70% 相关 | 人工标记相关/不相关 |
| 细纲检索 | 查询"感情线节拍"返回结果中至少 70% 相关 | 人工标记相关/不相关 |
| 章纲检索 | 查询"开局困境"返回结果中至少 80% 相关 | 人工标记相关/不相关 |
| 人物检索 | 查询"导师型人物"返回结果中至少 70% 相关 | 人工标记相关/不相关 |
| 事件检索 | 查询"雨中告别"返回结果中至少 80% 相关 | 人工标记相关/不相关 |

**验收标准设计原则**：
- P0 需求验收标准更严格（> 80%）
- P1 需求验收标准适中（> 70%）
- P2 需求验收标准宽松（> 60%）

### 3.4 运行健康与产物质量要求

| 要求 | 验收标准 | 度量方式 |
|---|---|---|
| 运行可诊断 | `full/continue` 产生结构化事件和统一报告，包含终态、阶段耗时/计数、API 与 Token、诊断及下一步 | 事件/报告契约测试与真实素材验收 |
| 产物可审计 | 确定性规则覆盖核心文件、章节覆盖、人物兜底档案、世界观和 insights 等信号 | 规则单元测试与审计汇总 |
| 审计只读 | 审计、可选复审和报告生成不修改事实文件，只允许在 `reports/` 新增报告 | 审计前后逐文件 SHA-256 对比 |
| 严重度可执行 | blocker 使运行失败并阻止同步；error 使运行降级；warning/info 不单独导致失败 | 编排与 CLI 退出码测试 |
| 报告可追溯 | 每个 run 保留不可变 YAML；`latest.yaml` 与 `latest.md` 原子更新 | 冲突、历史读取与原子写入测试 |
| 复审有预算 | 默认规则审计不调用 LLM；显式复审受时间和次数双预算约束 | reviewer/budget 测试与报告字段 |

这些要求解决“这次运行发生了什么、产物是否完整、下一步怎么处理”，不等同于检索结果质量提升。检索相关率、召回率和排序优劣仍必须通过人工 Golden Query 基线单独证明。

### 3.5 前置导航与人物素材完整性

前置导航 `evaluation.yaml` 的产品定位是给长篇素材提供全局方向信号：类型、前提、主线、阶段地图、核心人物候选、世界观维度、分析重点和采样覆盖范围。它帮助后续分析和人物提取阶段获得更稳定的全局参考，但不等同于章节滑动窗口；`--window` 只处理章级分析的相邻上下文。

人物素材分为两类：

- 完整小传：只覆盖自适应选择出的主要人物，包含人物弧线、心理动机、关键场景、关系和写作借鉴边界。
- 简档：覆盖其余达到候选阈值的人物，保留基础描述、出场、叙事功能和关系等信息。

这项能力的验收目标是让人物事实数据更完整、可审计，并为后续检索和写作 Agent 提供更可靠的上下文。它不构成人物检索相关率提升的证明；人物检索质量仍以 Golden Query 人工标注与评分为准。

### 分层世界观与作品画像

世界观事实数据采用分层结构表达：目录摘要、世界观概览、维度适用性、实体卡片和实体关系分离保存。实体必须具备稳定 ID、类型、名称、描述、重要性、首次出现、章节证据和置信度；关系必须引用实体 ID，并保留关系类型、描述、演化、证据和置信度。旧世界观四文件仍需可读、可同步、可搜索，读取时不得自动改写用户素材。

`work_profile.yaml` 是面向写作 Agent 的作品级入口，汇总作品钩子、读者期待、结构节奏、人物动力、世界观驱动、技法启示、证据索引、限制和置信度。它只做导航与创作策略摘要，不替代 `chapters.yaml`、人物档案、世界观实体等事实来源，不作为最终小说内容或检索质量评测。

storage、embedding 和 `search world` 必须通过统一读取器兼容新旧世界观。分层世界观结果可返回实体 ID、维度、证据、关键出场和关系摘要；但在 Golden Query 人工标注完成前，这只能说明结果结构更完整，不得声明世界观检索相关率、召回率或排序质量已经提升。

真实素材上的世界观重跑、作品画像生成、人物修复和完整流水线都可能调用 LLM 并修改事实文件；除只读 `validate artifacts` 和报告重建外，Agent 执行前必须获得用户明确授权。

## 四、6 个核心检索需求

### 需求 1：世界观检索

**触发时机**：场景 A — 前期规划

**用户意图**：找到类似类型/设定的世界观参考

**查询示例**：

| 查询风格 | 示例 | 用户状态 |
|---------|------|---------|
| **精确型** | "找修仙类小说的力量体系设计，分 5 个等级的" | 明确知道自己要什么 |
| **探索型** | "我想写一个科幻背景的修仙世界，有什么参考吗？" | 有想法但模糊 |
| **比较型** | "《凡人修仙传》和《仙逆》的境界体系有什么不同？" | 想对比多个方案 |
| **灵感型** | "有什么特别的势力设定？不要常见的宗门、帝国" | 寻求创意刺激 |
| **困境型** | "我设定的势力关系太简单了，怎么增加复杂度？" | 遇到具体问题 |
| **跨类型型** | "找有多个势力博弈的世界观，不限类型" | 类型不敏感 |
| **细节型** | "找地理空间复杂（多大陆/多星球）的设定" | 关注特定维度 |
| **反套路型** | "找打破'强者统治'逻辑的势力设计" | 想突破常规 |

**返回内容**：世界观实体、适用维度、证据章节、关键出场、关系摘要，以及力量体系、势力关系、地理空间或其他题材适用设定亮点

**需求特征**：

| 特征 | 说明 | 设计影响 |
|------|------|---------|
| 查询风格多样 | 精确型、探索型、比较型、灵感型、困境型等 | 检索需支持模糊匹配和精确匹配两种模式 |
| 单一关键词不足 | 用户意图表达模糊，需上下文理解 | Agent 需具备意图推断能力 |
| 多维度组合 | 世界观包含力量体系、势力、地理等 | 需支持 `--dimension` 参数组合查询 |
| 对比需求 | 用户需要对比多个方案的异同 | 返回多条结果时需展示对比视角 |
| 原文佐证 | 设定需原文细节验证 | 检索结果需关联章节号 |

---

### 需求 2：大纲检索

**触发时机**：场景 A — 前期规划

**用户意图**：找到类似类型/结构的大纲作为参考

**查询示例**：

| 查询风格 | 示例 | 用户状态 |
|---------|------|---------|
| **类型限定型** | "找 3 本修仙小说的大纲，主角都是废柴逆袭" | 类型明确 |
| **开局模式型** | "找开局是困境模式的小说大纲" | 关注起点设计 |
| **结构模式型** | "找多线叙事的玄幻大纲" | 关注叙事手法 |
| **节奏型** | "找前 50 章节奏紧凑的大纲，不要慢热" | 关注节奏控制 |
| **转折设计型** | "找中段有重大转折的大纲" | 关注转折时机 |
| **结局类型型** | "找开放式结局的大纲" | 关注终点设计 |
| **主题型** | "找复仇主题的大纲，看复仇弧线怎么设计" | 关注主题弧线 |
| **篇幅型** | "找 200 万字以上的大纲，看长篇怎么规划" | 关注篇幅管理 |
| **反套路型** | "找主角不是天才/废柴的常规人设定大纲" | 想突破套路 |
| **跨类型借鉴型** | "我想写都市商战，能参考悬疑的节奏设计吗？" | 跨类型学习 |

**返回内容**：完整的大纲结构树（幕 → 序列 → 节拍）

**需求特征**：

| 特征 | 说明 | 设计影响 |
|------|------|---------|
| 节奏参考需求 | 大纲的核心价值是理解节奏设计，而非套用模板 | 需展示幕→序列→节拍的层次结构 |
| 层次检索需求 | 用户可能关注整体结构、开局设计、转折设计等不同层次 | 需支持按层次展示不同粒度 |
| 节奏特征标注 | 用户需要知道每本大纲的节奏特点（慢热/紧凑） | 需存储节奏特征元数据 |
| 多方案对比 | 同一类型有多种大纲方案，用户需要对比异同 | 返回多条结果时需展示对比视角 |
| 章节验证需求 | 大纲需章节内容验证实际写法 | 需关联关键章节的摘要 |

### 需求 3：细纲检索

**触发时机**：场景 A — 前期规划

**用户意图**：找到类似情节推进模式的细纲

**查询示例**：

| 查询风格 | 示例 | 用户状态 |
|---------|------|---------|
| **推进模式型** | "找悬疑类小说的中段推进模式" | 关注特定阶段的推进 |
| **关系发展型** | "找感情线从相识到相爱的节拍设计" | 关注关系演进 |
| **铺垫型** | "找高潮前的铺垫序列" | 关注铺垫手法 |
| **困境突破型** | "找主角从低谷到反弹的节拍" | 关注困境-突破弧线 |
| **成长型** | "找主角第一次实力飞跃的推进设计" | 关注成长节奏 |
| **冲突升级型** | "找矛盾逐步激化的序列设计" | 关注冲突节奏 |
| **悬念型** | "找悬念铺设和揭晓的节奏设计" | 关注悬念管理 |
| **反转型** | "找有重大反转的序列，看反转怎么铺垫" | 关注反转手法 |
| **配角弧线型** | "找配角完整弧线的序列设计" | 关注配角叙事 |
| **群戏型** | "找多人物同时推进的序列设计" | 关注群戏调度 |

**返回内容**：序列级/节拍级的结构对比

**需求特征**：

| 特征 | 说明 | 设计影响 |
|------|------|---------|
| 节拍参考需求 | 细纲的核心价值是理解节拍设计（每个节拍的功能） | 需存储节拍功能说明 |
| 张力曲线关联 | 用户需要了解每个节拍的张力级别变化 | 需存储节拍张力值 |
| 推进节奏对比 | 同一推进目标可有不同节奏（快/慢） | 需支持按节奏类型筛选 |
| 篇幅适配需求 | 用户计划篇幅不同，需要不同长度的推进参考 | 需存储序列章节数 |

### 需求 4：章纲检索

**触发时机**：场景 B — 章节写作

**用户意图**：找到同类型章节的写法参考

**查询示例**：

| 查询风格 | 示例 | 用户状态 |
|---------|------|---------|
| **位置型** | "找 10 本小说第 1 章的开局写法" | 关注特定位置的章节 |
| **功能型** | "找转折点章节的写法" | 关注章节功能 |
| **张力型** | "找高张力章节（张力 ≥ 4）的写法" | 关注张力设计 |
| **情绪型** | "找压抑氛围的章节写法" | 关注情绪渲染 |
| **冲突型** | "找冲突爆发的章节写法" | 关注冲突处理 |
| **转折型** | "找信息反转章节的写法" | 关注反转手法 |
| **悬念型** | "找结尾留悬念的章节写法" | 关注悬念钩子 |
| **过渡型** | "找过渡章节的写法，不要太无聊" | 关注过渡技巧 |
| **群戏型** | "找多人物互动的章节写法" | 关注群戏处理 |
| **独角戏型** | "找主角独白/心理活动的章节写法" | 关注内心戏 |
| **场景型** | "找战斗场景的章节写法" | 关注场景类型 |
| **对话型** | "找对话驱动的章节写法" | 关注对话技巧 |
| **节奏型** | "找一章内完成完整弧线的写法" |关注章内节奏 |
| **铺垫型** | "找铺垫章节的写法，为后面的转折做准备" | 关注铺垫手法 |
| **闪回型** | "找使用闪回手法的章节写法" | 关注特殊手法 |

**返回内容**：章节摘要 + 章节功能标签 + 结构信息

**需求特征**：

| 特征 | 说明 | 设计影响 |
|------|------|---------|
| 功能参考需求 | 章节检索的核心是理解章节功能，而非抄写法 | 需存储章节功能标签 |
| 位置与功能区分 | 位置型查询（第 1 章）和功能型查询（转折章节）需不同策略 | 需支持按章节号和功能标签两种检索 |
| 章内结构需求 | 用户需要了解单章的内部结构（开场→核心→钩子） | 需存储章内结构分析（当前未实现） |
| 钩子设计需求 | 章节结尾的钩子设计是关键参考 | 需存储钩子类型（悬念/反转/情绪） |
| 前后章关联需求 | 单章理解需要前后章关联 | 需返回前后章节摘要链接 |

### 需求 5：人物检索

**触发时机**：场景 B — 人物写作

**用户意图**：找到同类人物的塑造参考

**查询示例**：

| 查询风格 | 示例 | 用户状态 |
|---------|------|---------|
| **原型型** | "找修仙类小说的导师型人物写法" | 关注人物原型 |
| **角色型** | "找反派角色的出场方式" | 关注角色定位 |

## 五、关键约束与原则

### 5.1 标注数据的定位

**标注数据是"检索参考"和"统计元数据"，不是"学习材料"。**

Agent 本身有能力：
- 从阅读过的海量文本中隐式学习模式
- 根据用户指令生成结构
- 参考具体样例调整输出

**不需要**从标注数据中"学习规律"来生成小说。标注数据的作用是让 Agent 能**检索到高质量参考样例**，然后糅合使用。

**统计元数据**（出场频率、张力分布、悬念章节数、章节功能分布等）是聚合信息，用于辅助 Agent 更准确地理解和检索，不是参考样例本身。

### 5.2 拆分悖论

**拆分越精细，边界越难定义；拆分越粗，规律提取越依赖理解。**

| 拆分层级 | 边界困难 | 规律提取困难 |
|----------|----------|--------------|
| 事件 | 戏剧动作模糊（不可控） | 需要理解事件角色 |
| 场景 | 焦点切换信号（半可控） | 需要理解场景关系 |
| 章节 | 无困难（完全可控） | 需要理解章节内容 |

**结论**：以章节为最小分析单元。边界完全可控，检索效果足够。

**例外——特殊章节类型**：小说中存在非叙事性章节（后记、番外、作者说等），系统需识别其类型并放宽分析要求。这些章节不参与张力评估、人物识别等叙事分析，但在检索时可作为过滤维度。

### 5.3 Agent 的核心能力

**Agent 有理解参考的能力，但需要统计数据辅助以保证准确性和完整性。**

例如，Agent 看到：
- 第一章摘要：入狱 → 困境 → 自救思路 → 能理解"开局范式"
- 第 1-5 章摘要：破案过程 → 能理解"推进范式"
- 第 7 章摘要：家人团聚 + 配角自杀悬念 → 能理解"转折范式"

但在千章级长篇小说中，仅靠摘要池难以准确识别高频人物和关键情节线。此时需要统计数据（出场频率、张力分布等）辅助 Agent 做出更完整的判断。

这些理解和统计辅助的结合，Agent 可以自己完成，不需要人为标注。

### 5.4 糅合机制

用户说的"把 N 条匹配度最高的参考糅合在一起"，具体含义是：

1. **检索**：Agent 根据用户需求找到 N 条最匹配的参考
2. **展示**：将 N 条参考以结构化形式展示给用户和 Agent
3. **理解**：Agent 分析每条参考的特点（结构、节奏、技法等）
4. **糅合**：结合用户的个性化需求，将多条参考的精华整合
5. **生成**：产出用户自己的大纲/细纲/章纲/人物/事件

**本项目负责的是第 1-2 步**（检索 + 展示）。第 3-5 步由 Agent 在写作场景中完成。

### 5.5 素材独立性原则

**每部小说的世界观、人物、事件相互独立，不跨书共享。**

检索时返回的是"参考样例"，不是"直接引用"。糅合后生成的内容属于用户自己的创作。

### 5.6 检索质量优先原则

**检索首先要找到真正有用、足够多样的参考，其次才优化响应速度。**

- 现有 4096 维向量是当前质量基线，不预设必须降维。
- 降维、近似向量索引或更换向量数据库，都必须先与质量基线对比。
- 性能优化不得以明显降低相关率、召回率或结果多样性为代价。
- 深度检索可以使用更大的候选集和更精细的重排，最长约 3 分钟是可接受的响应范围。
- 系统已提供 Golden Query、候选导出和评分框架；人工相关性标注尚未完成，质量结论继续延期。
- 质量模式采用中文词法、4096 维精确语义和结构化三路召回；单路失败必须在 trace 中说明。
- LLM 重排只有在人工对比证明不退化后才能默认启用；当前默认关闭。

## 七、不做的内容

基于上述需求，以下方向**明确不做**：

| 不做内容 | 原因 |
|----------|------|
| 事件拆分（"戏剧动作"级别） | 边界不可控，投入产出比极低 |
| 场景拆分 | 检索粒度与写作需求的 gap 应由 Agent 解决，不由素材库解决 |
| 人工或 LLM 精细标注结构角色 | 定义模糊，无法检验；当前只保留代码推断的粗粒度 `key_plot_point` |
| 精细张力曲线（逐章标注） | 无法检验；章节级张力评估（1-5）已用于大纲生成和检索排序，精度足够 |
| 从标注数据训练生成模型 | Agent 不需要训练，只需要参考 |
| 模板化自动生成 | 用户需要的是"糅合参考"，不是"模板套用" |
| 物品/怪物/能力独立检索 | 已由事件检索覆盖；随标签体系领域细分，未来可能成为独立维度 |
| Web UI | 所有操作通过 Agent CLI 和脚本完成 |

## 八、成功标准

本项目是否成功，取决于用户能否在实际写作中：

1. **找到高质量参考**：返回结果确实符合用户的题材、情节和写作功能需求
2. **避免遗漏与同质化**：候选范围足够广，最终结果来自多本小说和不同写法
3. **在可接受时间内完成**：深度检索最长约 3 分钟，且速度优化不突破质量下限
4. **方便地糅合**：返回的结构化数据让 Agent 能轻松分析糅合
5. **覆盖足够广**：500～5000 本小说提供持续扩展的参考池
6. **流水线具备断点续传能力**：中断后可从已完成章节继续，不重复分析
7. **单章失败不阻塞整体流程**：局部错误自动跳过，最终稳步进入可检索状态
8. **原文定位支持**：检索结果可追溯到原文具体位置，方便 Agent 精确定位
9. **运行结果可解释**：每次完整或续传运行都能从结构化日志与报告追溯阶段、资源消耗、诊断和终态
10. **产物质量可行动**：只读审计按稳定严重度给出问题证据和下一步，blocker 不得进入同步

**可度量指标**：
- 检索准确率：用户标记相关/不相关的比例 > 80%
- 检索召回率与排序质量：建立基准查询后，以 Recall@K、MRR、nDCG 等指标持续比较
- 结果多样性：同一小说不得长期垄断 Top N 结果
- 分析完整率：成功分析章节占比 > 95%
- 摘要信息密度：摘要长度 / 原文长度的比值维持在合理区间
- 运行报告完整率：具备完整终态事件的 `full/continue` 运行均可生成或重建报告
- 审计只读性：事实产物在审计与报告前后 SHA-256 保持不变

运行报告和产物审计用于提升可观测性与交付可信度，不作为“混合检索或重排优于 4096 维精确模式”的证据。

## 九、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| **中文网文排版格式极其多样** | 大量小说入库失败，无法达成 500～5000 本规模目标 | 废弃死板的纯正则，引入智能试探切分与预清洗机制，兼容各种中文数字与卷章嵌套。对特殊章节类型（课程表、作者说等）做正则防护 |
| **LLM 调用成本控制与限流崩溃** | 并发受限，程序直接崩溃，造成巨额 Token 浪费 | 必须建立带有指数退避重试（Exponential Backoff）和断点续传能力的健壮调度层，严禁裸调 API。支持多 LLM 服务商并行执行，规避单一服务商限速/故障 |
| **超长文本导致 LLM 上下文截断** | 丢失核心转折和章节悬念，导致大纲/章纲严重失真 | 基于 Token 计算的动态摘要策略，首尾剧情完整输入。模型支持 1M 上下文后，以实际 tokens 而非字数为截断依据 |
| LLM 章级分析质量不稳定 | 检索精度下降 | 三层防御：动态提示词强度递增 + 动态温度调节 + 输出相似度检测。前期使用 thinking 模式保证质量，后期关闭 thinking 启用动态温度防止模式化输出 |
| LLM 批量输出 JSON 不完整 | 频繁降级为逐章模式，入库耗时大幅增加 | 降级时记录原因日志便于诊断；兼容 LLM 返回 list 类型；错误信息禁止写入数据文件 |
| 向量搜索语义不准 | 匹配度低 | 选择合适的 embedding 模型，针对中文优化 |
| 数据规模增长超预期 | 性能下降 | 保留 4096 维质量基线并做分级容量压测；只有质量评测和性能基准共同证明必要时，才采用近似索引、降维或独立向量数据库 |
| 检索结果同质化 | 参考价值低 | 引入多样性排序（不同类型/作者分散） |
| 标签体系超出 LLM 上下文 | 标签截断，LLM 无法使用完整标签字典 | 标签分级架构（L0-L4），按题材领域动态加载对应子集 |
