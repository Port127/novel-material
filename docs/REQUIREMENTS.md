## 一、项目定位

**novel-material 是一个小说写作参考检索库。**

它不是用来"学习规律"的训练系统，而是用来**提供检索参考**的素材库。

**类比**：建筑师不会从 100 栋建筑中"学习"如何设计，而是翻阅优秀案例图集，把多个案例的精华组合到自己的设计中。本项目就是那个"案例图集"。

## 二、用户的两个写作场景

### 场景 A：前期规划阶段

用户在动笔之前，需要构建小说的整体框架。

**需要辅助生成的内容：**
- 世界观设定
- 规则体系（力量体系、社会规则等）
- 大纲（全书结构）
- 细纲（幕/序列/节拍级结构）

**用户工作流：**
1. 用户与 Agent 讨论，明确自己想要什么样的世界观/大纲
2. Agent 从本项目中检索 N 条匹配度最高的参考
3. 用户 + Agent 将多条参考糅合，形成自己的设定

### 场景 B：章节写作阶段

用户在写具体章节时，需要参考同类章节的写法。

**需要辅助生成的内容：**
- 章纲（单章节的结构和推进方式）
- 人物描写（人物的出场、互动、弧光展现）
- 事件描写（特定类型事件的写法）

**用户工作流：**
1. 用户写到某个章节/人物/事件时，与 Agent 讨论需要什么参考
2. Agent 从本项目中检索 N 条匹配度最高的参考
3. 用户 + Agent 将多条参考糅合，写出自己的内容

## 三、6 个核心检索需求

### 需求 1：世界观检索

**触发时机**：场景 A — 前期规划

**用户意图**：找到类似类型/设定的世界观参考

**查询示例**：
- "找修仙类小说的力量体系设计，分 5 个等级的"
- "找有多个势力博弈的世界观"
- "找地理空间复杂（多大陆/多星球）的设定"

**返回内容**：力量体系结构 + 势力关系图 + 地理空间描述 + 设定亮点

### 需求 2：大纲检索

**触发时机**：场景 A — 前期规划

**用户意图**：找到类似类型/结构的大纲作为参考

**查询示例**：
- "找 3 本修仙小说的大纲，主角都是废柴逆袭"
- "找开局是困境模式的小说大纲"
- "找多线叙事的玄幻大纲"

**返回内容**：完整的大纲结构树（幕 → 序列 → 节拍）

### 需求 3：细纲检索

**触发时机**：场景 A — 前期规划

**用户意图**：找到类似情节推进模式的细纲

**查询示例**：
- "找悬疑类小说的中段推进模式"
- "找感情线从相识到相爱的节拍设计"
- "找高潮前的铺垫序列"

**返回内容**：序列级/节拍级的结构对比

### 需求 4：章纲检索

**触发时机**：场景 B — 章节写作

**用户意图**：找到同类型章节的写法参考

**查询示例**：
- "找 10 本小说第 1 章的开局写法"
- "找高潮章节的写法"
- "找转折点章节的写法"

**返回内容**：章节摘要 + 章节功能标签 + 结构信息

### 需求 5：人物检索

**触发时机**：场景 B — 人物写作

**用户意图**：找到同类人物的塑造参考

**查询示例**：
- "找修仙类小说的导师型人物写法"
- "找反派角色的出场方式"
- "找配角的人物弧光设计"

**返回内容**：人物小传 + 关键出场章节 + 互动模式

### 需求 6：事件检索

**触发时机**：场景 B — 事件写作

**用户意图**：找到同类事件的写法参考

**查询示例**：
- "找雨中告别的写法"
- "找主角初次突破的写法"
- "找团队内讧的写法"

**返回内容**：匹配的章节摘要 + 上下文信息

## 四、关键约束与原则

### 4.1 标注数据的定位

**标注数据是"检索参考"和"统计元数据"，不是"学习材料"。**

Agent 本身有能力：
- 从阅读过的海量文本中隐式学习模式
- 根据用户指令生成结构
- 参考具体样例调整输出

**不需要**从标注数据中"学习规律"来生成小说。标注数据的作用是让 Agent 能**检索到高质量参考样例**，然后糅合使用。

**统计元数据**（出场频率、张力分布、悬念章节数、章节功能分布等）是聚合信息，用于辅助 Agent 更准确地理解和检索，不是参考样例本身。

### 4.2 拆分悖论

**拆分越精细，边界越难定义；拆分越粗，规律提取越依赖理解。**

| 拆分层级 | 边界困难 | 规律提取困难 |
|----------|----------|--------------|
| 事件 | 戏剧动作模糊（不可控） | 需要理解事件角色 |
| 场景 | 焦点切换信号（半可控） | 需要理解场景关系 |
| 章节 | 无困难（完全可控） | 需要理解章节内容 |

**结论**：以章节为最小分析单元。边界完全可控，检索效果足够。

**例外——特殊章节类型**：小说中存在非叙事性章节（后记、番外、作者说等），系统需识别其类型并放宽分析要求。这些章节不参与张力评估、人物识别等叙事分析，但在检索时可作为过滤维度。

### 4.3 Agent 的核心能力

**Agent 有理解参考的能力，但需要统计数据辅助以保证准确性和完整性。**

例如，Agent 看到：
- 第一章摘要：入狱 → 困境 → 自救思路 → 能理解"开局范式"
- 第 1-5 章摘要：破案过程 → 能理解"推进范式"
- 第 7 章摘要：家人团聚 + 配角自杀悬念 → 能理解"转折范式"

但在千章级长篇小说中，仅靠摘要池难以准确识别高频人物和关键情节线。此时需要统计数据（出场频率、张力分布等）辅助 Agent 做出更完整的判断。

这些理解和统计辅助的结合，Agent 可以自己完成，不需要人为标注。

### 4.4 糅合机制

用户说的"把 N 条匹配度最高的参考糅合在一起"，具体含义是：

1. **检索**：Agent 根据用户需求找到 N 条最匹配的参考
2. **展示**：将 N 条参考以结构化形式展示给用户和 Agent
3. **理解**：Agent 分析每条参考的特点（结构、节奏、技法等）
4. **糅合**：结合用户的个性化需求，将多条参考的精华整合
5. **生成**：产出用户自己的大纲/细纲/章纲/人物/事件

**本项目负责的是第 1-2 步**（检索 + 展示）。第 3-5 步由 Agent 在写作场景中完成。

### 4.5 素材独立性原则

**每部小说的世界观、人物、事件相互独立，不跨书共享。**

检索时返回的是"参考样例"，不是"直接引用"。糅合后生成的内容属于用户自己的创作。

## 五、规模目标

| 指标 | 数值 |
|------|------|
| 小说数量 | 1000 本起步 |
| 每本章节 | ~500 章 |
| 总章节数 | ~50 万章 |
| 总记录数（含标签、人物出场等） | 千万级 |
| 检索响应时间 | < 2 秒 |
| 并发检索 | 支持多 Agent 同时调用 |
| 新书入库到可检索（100 章以内） | < 30 分钟（含 LLM 分析） |
| 新书入库到可检索（1000 章） | < 3 小时（含 LLM 分析） |

## 六、不做的内容

基于上述需求，以下方向**明确不做**：

| 不做内容 | 原因 |
|----------|------|
| 事件拆分（"戏剧动作"级别） | 边界不可控，投入产出比极低 |
| 场景拆分 | 检索粒度与写作需求的 gap 应由 Agent 解决，不由素材库解决 |
| 结构角色标注（转折/高潮定义） | 定义模糊，无法检验 |
| 精细张力曲线（逐章标注） | 无法检验；章节级张力评估（1-5）已用于大纲生成和检索排序，精度足够 |
| 从标注数据训练生成模型 | Agent 不需要训练，只需要参考 |
| 模板化自动生成 | 用户需要的是"糅合参考"，不是"模板套用" |
| 物品/怪物/能力独立检索 | 已由事件检索覆盖；随标签体系领域细分，未来可能成为独立维度 |
| Web UI | 所有操作通过 Agent CLI 和脚本完成 |

## 七、成功标准

本项目是否成功，取决于用户能否在实际写作中：

1. **快速找到参考**：输入需求描述后，2 秒内返回匹配结果
2. **找到高质量参考**：返回的结果确实是用户需要的类型
3. **方便地糅合**：返回的结构化数据让 Agent 能轻松分析糅合
4. **覆盖足够广**：1000 本小说提供充足的参考池
5. **流水线具备断点续传能力**：中断后可从已完成章节继续，不重复分析
6. **单章失败不阻塞整体流程**：局部错误自动跳过，最终稳步进入可检索状态
7. **原文定位支持**：检索结果可追溯到原文具体位置，方便 Agent 精确定位

**可度量指标**：
- 检索准确率：用户标记相关/不相关的比例 > 80%
- 分析完整率：成功分析章节占比 > 95%
- 摘要信息密度：摘要长度 / 原文长度的比值维持在合理区间

## 八、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| **中文网文排版格式极其多样** | 大量小说入库失败，无法达成”1000本”规模目标 | 废弃死板的纯正则，引入智能试探切分与预清洗机制，兼容各种中文数字与卷章嵌套。对特殊章节类型（课程表、作者说等）做正则防护 |
| **LLM 调用成本控制与限流崩溃** | 并发受限，程序直接崩溃，造成巨额 Token 浪费 | 必须建立带有指数退避重试（Exponential Backoff）和断点续传能力的健壮调度层，严禁裸调 API。支持多 LLM 服务商并行执行，规避单一服务商限速/故障 |
| **超长文本导致 LLM 上下文截断** | 丢失核心转折和章节悬念，导致大纲/章纲严重失真 | 基于 Token 计算的动态摘要策略，首尾剧情完整输入。模型支持 1M 上下文后，以实际 tokens 而非字数为截断依据 |
| LLM 章级分析质量不稳定 | 检索精度下降 | 三层防御：动态提示词强度递增 + 动态温度调节 + 输出相似度检测。前期使用 thinking 模式保证质量，后期关闭 thinking 启用动态温度防止模式化输出 |
| LLM 批量输出 JSON 不完整 | 频繁降级为逐章模式，入库耗时大幅增加 | 降级时记录原因日志便于诊断；兼容 LLM 返回 list 类型；错误信息禁止写入数据文件 |
| 向量搜索语义不准 | 匹配度低 | 选择合适的 embedding 模型，针对中文优化 |
| 数据规模增长超预期 | 性能下降 | 定期评估 pgvector → Qdrant 迁移必要性。长篇小说（1000+ 章）按章独立存储 + 惰性加载，避免单文件成为性能瓶颈 |
| 检索结果同质化 | 参考价值低 | 引入多样性排序（不同类型/作者分散） |
| 标签体系超出 LLM 上下文 | 标签截断，LLM 无法使用完整标签字典 | 标签分级架构（L0-L4），按题材领域动态加载对应子集 |
