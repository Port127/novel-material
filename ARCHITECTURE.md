# ARCHITECTURE.md

## 项目定位

小说写作参考检索库，服务于两个写作场景：
- **前期规划**：世界观、大纲、细纲检索
- **章节写作**：章纲、人物、事件检索

## 核心设计原则

1. **标注数据是检索参考，不是学习材料** — Agent 自行理解模式，本项目提供高质量参考
2. **以章节为最小分析单元** — 放弃事件/场景拆分（边界不可控，投入产出比低）
3. **YAML 是 Source of Truth** — 数据库是派生查询层，便于版本控制和迁移
4. **渐进式架构** — SQLite 验证 → PostgreSQL + pgvector → 按需升级

## 系统架构

```
用户检索请求
     |
     v
┌─────────────────┐
│   检索路由层      │  分析查询意图，路由到具体脚本
└─────────────────┘
         |
   ┌─────┴─────┐
   |           |
结构化查询     语义查询
   |           |
   v           v
┌──────┐   ┌──────────┐
│PostgreSQL│ │ pgvector  │
└──────┘   └──────────┘
   |           |
   └─────┬─────┘
         |
         v
┌─────────────────┐
│   融合排序层      │  综合结构化+语义结果
└─────────────────┘
```

## 处理流程

```
原文 → 格式清洗 → 章节切分 → 大纲 → 世界观 → 人物 → 标签 → 章级分析 → Embedding → 同步数据库
```

## 技术栈

| 组件 | 技术 | 说明 |
|------|------|------|
| 数据库 | PostgreSQL + pgvector | 结构化查询 + 向量搜索 |
| 向量维度 | 1024 | BGE-large-zh（中文优化） |
| Source of Truth | YAML 文件 | 版本控制友好 |
| 脚本语言 | Python 3.10+ | 统一工具链 |

## 数据规模

| 指标 | 数值 |
|------|------|
| 小说数量 | 1000 本起步 |
| 每本章节 | ~500 章 |
| 总章节数 | ~50 万章 |
| 总记录数 | 千万级 |
| 检索响应时间 | < 2 秒 |

## Anti-Patterns

1. **不拆分事件** — 边界不可控，LLM 理解偏差无法检验
2. **不标注结构角色** — 转折/高潮定义模糊，Agent 可自行理解
3. **不单独检索物品/怪物/能力** — 由事件检索覆盖
4. **不模板化生成** — 用户需要"糅合参考"，不是"模板套用"
